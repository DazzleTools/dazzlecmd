# Human Test Checklist: v0.7.20 -- Setup Parity + Venv-Per-Tool Dispatch

**Target version:** v0.7.20
**Ships with commit:** TBD (v0.7.20)
**Companion automated tests:** 102 new tests across `test_setup_resolve.py` (25), `test_python_runner_interpreter.py` (15), `test_venv_integration.py` (4), `test_templates.py` (40), `test_vars_integration.py` (18). Full suite: 616 passing, 6 platform-skipped.

## What this is

v0.7.20 does four things:

1. Extends the `setup.platforms` schema to match `runtime.platforms` (nested subtype dicts + `general` fallback, via the shared `platform_resolve` module). Flat-string form retained as shorthand.
2. Adds `_schema_version` checking on the setup block.
3. Closes 4b.3 (#22): `make_python_runner` honors `runtime.interpreter`, enabling the venv-per-tool pattern. A synthetic fixture at `tests/fixtures/venv_exercise/` validates the end-to-end flow with 7 heavy deps (numpy/pandas/etc.).
4. Adds template variables (`_vars`, issue #41): declare shared command fragments at manifest-top, block, platform, or subtype level; reference via `{{name}}`. Four-tier scope with dynamic lookup; nested var references with cycle detection; unresolved references hard-error.

Automated tests cover the module behavior and one full integration test. This checklist is the human-eyeball pass: does `dz setup` actually render sensibly, does the venv pattern feel right when running it by hand, does error messaging on a malformed setup block help authors.

## High-Value Verification (~8 minutes)

### HV.1 -- `dz setup` still works on existing flat-string schema

Validates: backwards compatibility. No in-repo tool currently uses `setup.platforms` at all, so we scaffold one.

- [ ] **One-time setup for all v0.7.20 HV scenarios**: point `DAZZLECMD_CONFIG` at a temp file, then use the manual-registration pattern below. `dz kit add <path>` does NOT accept local paths (git URLs only); the tester sweep confirmed this.

  **cmd.exe**: `set DAZZLECMD_CONFIG=%TEMP%\dz_v0720_checklist.json`
  **PowerShell**: `$env:DAZZLECMD_CONFIG = "$env:TEMP\dz_v0720_checklist.json"`
  **POSIX**: `export DAZZLECMD_CONFIG=/tmp/dz_v0720_checklist.json`

  Manual kit registration (once):

  1. Create a directory `test_runs/v0.7.20/` that will hold all scratch tools.
  2. Create `test_runs/v0.7.20/.kit.json`:
     ```json
     {"name": "v0-7-20-scratch", "tools_dir": ".", "manifest": ".dazzlecmd.json"}
     ```
  3. Activate via the config: either edit `DAZZLECMD_CONFIG` to include `"active_kits": ["v0-7-20-scratch"]`, OR run `dz kit enable v0-7-20-scratch` AFTER adding the kit path to `dz`'s known-kits list (see your installation's kit-registration mechanism).

  **IMPORTANT**: each tool manifest's `"namespace"` must match the kit name (`v0-7-20-scratch`). Mismatched namespace makes the tool invisible (`_kit_active = False`).

- [ ] **Setup**: create `test_runs/v0.7.20/flat-setup/.dazzlecmd.json`:
  ```json
  {
      "name": "flat-setup-demo",
      "namespace": "v0-7-20-scratch",
      "runtime": {"type": "python", "script_path": "demo.py"},
      "setup": {
          "command": "echo default-install",
          "platforms": {
              "windows": "echo windows-install",
              "linux": "echo linux-install",
              "macos": "echo macos-install"
          }
      }
  }
  ```
  Create `test_runs/v0.7.20/.kit.json`:
  ```json
  {"name": "v0-7-20-scratch", "tools_dir": ".", "manifest": ".dazzlecmd.json"}
  ```
  Register manually via isolated config (see v0.7.19 HV.2 notes).

- [ ] **Steps** (same across all shells):
  ```
  dz setup flat-setup-demo
  ```

- [ ] **Expected**:
  - stdout shows `Command: echo <your-platform>-install` (matching your current OS)
  - Echo output matches (windows-install / linux-install / macos-install)
  - Exit 0

- [ ] **Why this matters**: if flat-string normalization breaks, every future author
  who uses the shorthand gets a cryptic error. This test confirms the fallback path.

### HV.2 -- `dz setup` works on new nested-dict schema with subtypes

Validates: the new canonical form resolves correctly with subtype fallback.

- [ ] **Setup**: create `test_runs/v0.7.20/nested-setup/.dazzlecmd.json`:
  ```json
  {
      "name": "nested-setup-demo",
      "namespace": "v0-7-20-scratch",
      "runtime": {"type": "python", "script_path": "demo.py"},
      "setup": {
          "platforms": {
              "linux": {
                  "debian": {"command": "echo linux-debian-install"},
                  "rhel":   {"command": "echo linux-rhel-install"},
                  "general":{"command": "echo linux-generic-install"}
              },
              "windows": {"command": "echo windows-install"},
              "macos":   {"command": "echo macos-install"}
          }
      }
  }
  ```

- [ ] **Steps**:
  ```
  dz setup nested-setup-demo
  ```

- [ ] **Expected**:
  - On Linux Debian/Ubuntu: output contains `linux-debian-install`
  - On Linux RHEL/Fedora: output contains `linux-rhel-install`
  - On other Linux (Arch, NixOS, etc.): output contains `linux-generic-install`
  - On Windows: output contains `windows-install`
  - On macOS: output contains `macos-install`
  - Exit 0

- [ ] **Why this matters**: subtype fallback is the raison d'etre of the nested
  schema. If Debian doesn't get its Debian branch, authors will stop using
  subtypes and we'll be back to "write shell-if-else inside a string."

### HV.3 -- `dz setup` error message when no platform matches is actionable

Validates: the error path when a manifest's setup has no applicable command for the current host.

- [ ] **Setup**: create a manifest whose `setup.platforms` includes only OSes
  you're NOT running on. On Linux, make it Windows/macOS only:
  ```json
  {
      "name": "no-platform-match",
      "namespace": "v0-7-20-scratch",
      "runtime": {"type": "python", "script_path": "demo.py"},
      "setup": {
          "platforms": {
              "windows": "echo win",
              "macos": "echo mac"
          }
      }
  }
  ```
  (Omit Linux; no top-level `command` either.)

- [ ] **Steps** (on Linux):
  ```
  dz setup no-platform-match
  ```

- [ ] **Expected**:
  - stderr reads something like `No setup command available for platform 'linux.debian'`
  - Message suggests adding `setup.command`, `setup.platforms.linux`, or `setup.platforms.linux.general`
  - Exit 1

- [ ] **Why this matters**: users hitting this error shouldn't have to read source.
  The hint should tell them exactly which manifest key to add.

### HV.6 -- Template variables (`_vars`) end-to-end

Validates: the `_vars` substitution feature works on a real tool manifest with the shared-venv-path pattern (the primary use case from issue #41).

- [ ] **Setup**: create `test_runs/v0.7.20/vars-demo/.dazzlecmd.json`:
  ```json
  {
      "name": "vars-demo",
      "namespace": "v0-7-20-scratch",
      "_vars": {
          "venv_dir": ".venv",
          "venv_bin": "{{venv_dir}}/bin",
          "venv_scripts": "{{venv_dir}}\\Scripts"
      },
      "runtime": {
          "type": "python",
          "script_path": "demo.py",
          "platforms": {
              "linux":   {"interpreter": "{{venv_bin}}/python"},
              "windows": {"interpreter": "{{venv_scripts}}\\python.exe"}
          }
      },
      "setup": {
          "platforms": {
              "linux":   {"command": "python3 -m venv {{venv_dir}} && {{venv_bin}}/pip install -U pip"},
              "windows": {"command": "python -m venv {{venv_dir}} && {{venv_scripts}}\\pip install -U pip"}
          }
      }
  }
  ```

- [ ] **Steps**:
  ```
  dz info vars-demo
  dz info vars-demo --raw
  ```

- [ ] **Expected** for `dz info vars-demo` (default, substituted):
  - `Runtime:     python  (resolved for <os>.<subtype>)`
  - On Linux: `Interpreter: .venv/bin/python` (substituted, no `{{...}}` visible)
  - On Windows: `Interpreter: .venv\Scripts\python.exe` (substituted, backslashes)

- [ ] **Expected** for `dz info vars-demo --raw`:
  - `Runtime:     python  (raw, unresolved)`
  - Output shows the `_vars` block as-declared (`venv_dir: .venv`, `venv_bin: {{venv_dir}}/bin`, ...)
  - Platform interpreters shown with `{{...}}` markers unresolved

- [ ] **Error case**: add a manifest with `{{undefined_var}}` and no matching `_vars` entry. Run `dz info <tool>`:
  - Expected: error message includes `UnresolvedTemplateVariableError` with the var name and list of available vars at the error site.

- [ ] **Why this matters**: `_vars` is the main DRY-up mechanism for cross-platform manifests. Authors will use this extensively once they see the pattern in the docs. A regression here would silently duplicate paths across every tool author's manifests; error-message quality determines how quickly they diagnose typos.

## What the automated tests DON'T cover

- **Shell rendering on Windows codepage** for `dz setup` output when setup commands contain special characters.
- **Real subprocess behavior** when setup command itself fails mid-way (partial install leaves a broken venv). Checklist doesn't currently probe this; authors who hit it will need to `dz safedel .venv` manually.
- **Interactive prompts** (e.g., sudo password on Linux setup commands) -- subprocess.run with shell=True + no stdin may hang. Not an issue for the current fixture (no sudo).
- **Cross-platform portability** of setup commands written by authors -- the engine faithfully dispatches; shell-correctness is the author's responsibility.
- **Error message quality** when a setup block has a malformed `_schema_version`. Covered by automated tests via UnsupportedSchemaVersionError, but author experience should be eyeballed.
