# Human Test Checklist: Phase 4c.4 -- Docker Runtime + Registry Isolation + `dz setup` Listing

**Target version:** v0.7.21
**Ships with commit:** TBD (Phase 4c.4)
**Companion automated tests:** 44 new tests across 4 files: `test_docker_runner.py` (19 mocked unit tests), `test_docker_integration.py` (8 real-Docker integration tests, opt-in via `@pytest.mark.docker_integration`), `test_cli_info.py` (+7 tests for Docker field rendering), `test_cli_setup.py` (+9 tests for listing mode polish). Full suite: 668 passing (+43 vs v0.7.20 baseline of 625), 6 platform-skipped.

## What this is

v0.7.21 ships four concurrent pieces:

1. **Docker runtime (Phase 4c.4)** -- new `runtime.type: "docker"` dispatches via `docker run` with manifest-declared image, volumes, env, env_passthrough, docker_args, and informational inner_runtime. Pre-flight `docker images -q` check with actionable `dz setup <fqcn>` hint on miss.
2. **Registry test isolation (Phase 4c.6)** -- `RunnerRegistry.reset()` classmethod + autouse conftest fixture prevent test pollution when tests register extension runtime types.
3. **`dz setup` no-arg listing mode polish (#33 tail)** -- lists tools with `setup.command` OR `setup.platforms.*` declared; alphabetized by FQCN; dynamic column width.
4. **`docs/guides/dz-setup.md`** -- new CLI reference (mirrors dz-kit.md / dz-tree.md pattern).

Automated tests cover unit and integration behavior. This checklist covers the human-eyeball surface: actual `docker run` invocations, `dz info` rendering, listing output aesthetics, and error message quality.

### HV.2 -- `dz info` renders Docker-specific fields

Validates: authors inspecting a Docker-type manifest see all relevant fields.

- [ ] **Steps** (same across all shells, uses the committed fixture):
  ```
  cd tests/fixtures/docker_tool
  ```

- [ ] **Expected** for `--raw` view:
  - `Runtime:     docker  (raw, unresolved)`
  - `Image:       {{image_name}}:{{image_tag}}` -- unresolved template markers visible
  - `Env:         1 var(s)` followed by `DAZZLECMD_DOCKER_TEST_EXPLICIT_ENV=from-manifest-env-dict`
  - `Env passthru: DAZZLECMD_DOCKER_TEST_ENV1, DAZZLECMD_DOCKER_TEST_ENV2` (names only; no values)
  - `Docker args: --rm`
  - `Inner runtime: (informational) type=python, interpreter=python3, script=/app/app.py`
  - `_vars (manifest-top):` with `image_name` and `image_tag` values

- [ ] **Expected** for default (resolved):
  - `Runtime:     docker  (resolved for <os>.<subtype>)` on current host
  - `Image:       dazzlecmd-test-docker-tool:v1` -- SUBSTITUTED (no `{{...}}` visible)
  - Trailing advisory line: `(manifest declares conditional dispatch; use --raw to see the full declaration)` -- expected, informational

- [ ] **Why this matters**: `dz info --raw` is the author's debugging surface for Docker manifests. If a field is missing or misformatted, authors can't inspect their own configurations.

### HV.3 -- `dz setup` listing mode with dummy scratch kit

Validates: `dz setup` (no args) lists tools with setup declared; sorted alphabetically; dynamic column width; handles both flat and nested setup forms.

- [ ] **Setup** (self-contained -- differs from the v0.7.19 pattern because v0.7.21's loader only discovers kits inside the in-repo `kits/` directory):

  1. Isolate config: set `DAZZLECMD_CONFIG` to a temp file (see v0.7.19 HV.2 notes for the cross-shell commands).

  2. Create a scratch kit directory INSIDE `projects/`:
     ```
     mkdir -p projects/hv3-scratch
     ```

  3. Create `projects/hv3-scratch/a-flat-setup/.dazzlecmd.json`:
     ```json
     {
         "name": "a-flat-setup",
         "namespace": "hv3-scratch",
         "description": "HV.3 flat setup test tool",
         "runtime": {"type": "python", "script_path": "demo.py"},
         "setup": {"command": "echo a", "note": "Flat setup test"}
     }
     ```

  4. Create `projects/hv3-scratch/b-nested-only/.dazzlecmd.json`:
     ```json
     {
         "name": "b-nested-only",
         "namespace": "hv3-scratch",
         "description": "HV.3 platforms-only setup (no top-level command)",
         "runtime": {"type": "python", "script_path": "demo.py"},
         "setup": {"platforms": {"linux": {"command": "echo linux"}, "windows": {"command": "echo win"}}}
     }
     ```

  5. Create `projects/hv3-scratch/c-no-setup/.dazzlecmd.json`:
     ```json
     {
         "name": "c-no-setup",
         "namespace": "hv3-scratch",
         "description": "HV.3 no setup block",
         "runtime": {"type": "python", "script_path": "demo.py"}
     }
     ```

  6. Create `kits/hv3-scratch.kit.json`:
     ```json
     {
         "name": "hv3-scratch",
         "tools_dir": "projects/hv3-scratch",
         "manifest": ".dazzlecmd.json",
         "tools": ["hv3-scratch:a-flat-setup", "hv3-scratch:b-nested-only", "hv3-scratch:c-no-setup"],
         "description": "v0.7.21 HV.3 scratch"
     }
     ```

  7. Activate in config:
     ```json
     {"active_kits": ["core", "hv3-scratch"]}
     ```

  **NOTE**: the v0.7.19 HV.2 pattern puts scratch kits under `test_runs/`. That doesn't work in v0.7.21 because the loader's `discover_kits` only scans the in-repo `kits/` directory. Scratch kits must live inside `projects/` + be registered via `kits/<name>.kit.json`. Confirmed by v0.7.21 tester sweep.

- [ ] **Steps** (cross-shell, uses `dz` command only):
  ```
  dz setup
  ```

- [ ] **Expected**:
  - Output shows 2 tools, alphabetically sorted: `a-flat-setup` then `b-nested-only`
  - `c-no-setup` does NOT appear
  - Each line formatted `  <fqcn>  <note or dash>`
  - Footer reads `Run: dz setup <tool> to execute a tool's setup.`
  - Exit code 0

- [ ] **Why this matters**: pre-v0.7.21 listing missed tools with only `setup.platforms.*` (no top-level command). The fix catches these; verify end-to-end.

### HV.4 -- Docker pre-flight error when image missing

Validates: the pre-flight check surfaces the `dz setup` hint cleanly when the image isn't present.

- [ ] **Setup**: ensure the image is NOT present:
  ```
  docker rmi dazzlecmd-test-docker-tool:v1
  ```

- [ ] **Steps** (using pytest-direct invocation because the runner uses inherited stdio):
  ```
  python -c "
  from dazzlecmd_lib.registry import make_docker_runner
  runner = make_docker_runner({
      'name': 'docker-test-tool',
      '_fqcn': 'test:docker-test-tool',
      '_dir': 'tests/fixtures/docker_tool',
      'runtime': {'type': 'docker', 'image': 'dazzlecmd-test-docker-tool:v1'}
  })
  import sys
  sys.exit(runner([]))
  "
  ```

- [ ] **Expected**:
  - stderr reads: `Error: Docker image 'dazzlecmd-test-docker-tool:v1' not found locally.` followed by `       Try: dz setup test:docker-test-tool`
  - Exit code 1
  - No Python traceback

- [ ] **Why this matters**: users hitting missing-image need actionable guidance. Generic "image not found" isn't enough; the `dz setup` hint tells them exactly what to do.

### HV.5 -- Integration test image ships a real container that isolates from host

Validates: containers are actually containerized (not running on the host via passthrough).

- [ ] **Setup**: ensure the image is present:
  ```
  pytest tests/test_docker_integration.py::TestDockerIntegration::test_image_build_succeeded -v -m docker_integration
  ```

- [ ] **Steps**:
  ```
  docker run --rm dazzlecmd-test-docker-tool:v1 arg1 arg2 arg3
  ```

- [ ] **Expected** (these 5 fields + additional env/volume diagnostic lines):
  - Output contains `DAZZLECMD_DOCKER_TEST_SIGNATURE=v1`
  - `DAZZLECMD_DOCKER_TEST_HOSTNAME=<random hash>` -- NOT your host's hostname
  - `DAZZLECMD_DOCKER_TEST_UNAME_SYSTEM=Linux` -- even if your host is Mac/Windows
  - `DAZZLECMD_DOCKER_TEST_PYTHON_VERSION=3.11.X` -- alpine-python, likely differs from host
  - `DAZZLECMD_DOCKER_TEST_ARGV_COUNT=3` with indexed args
  - Exit code 0

  Actual output contains ~13 lines including `DAZZLECMD_DOCKER_TEST_ENV:*` entries for each declared-watched env var and `DAZZLECMD_DOCKER_TEST_VOLUME_CONTENT` probe. The 5 fields above are the signal lines; the rest is diagnostic and NOT part of the validation criteria.

- [ ] **Why this matters**: confirms the container genuinely isolates from the host. If hostname matched or kernel info matched the host, something would be wrong with the container runtime setup.

## What the automated tests DON'T cover

- **Terminal rendering of multi-line `dz info` output** on cramped narrow terminals -- mocks assert substrings, not column wrapping.
- **`docker run` interactive mode** (`-it`) with a TTY attached. Non-interactive dispatch is tested; interactive behavior requires a human with a real terminal.
- **Cross-platform volume path handling** (Docker Desktop on Windows/WSL, rootless Podman, Colima, Rancher Desktop, OrbStack). All use the docker CLI so basic dispatch works; edge cases (volume permissions, mount translation) vary per engine.
- **Error message quality when `docker run` fails mid-execution** (image exists but ENTRYPOINT crashes). Integration tests check exit-code propagation but not the human-readability of container stderr.
- **docker compose / docker stack** -- out of v0.7.21 scope.
- **Podman / nerdctl variants** -- work via PATH alias but not explicitly tested.
