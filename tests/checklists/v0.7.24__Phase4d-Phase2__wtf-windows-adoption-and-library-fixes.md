# Human Test Checklist: v0.7.24 -- wtf-windows Adoption and Library Fixes (Phase 4d, Phase 2+3)

**Target version:** v0.7.24 (dazzlecmd-lib) + wtf-windows adoption commit
**Ships with commit:** TBD (coordinated pair: dazzlecmd v0.7.24 + wtf-windows repo commit)
**Companion automated tests:** 4 new regression tests across 2 files. Full dazzlecmd suite: 844 passing, 6 platform-skipped (up from 842 at v0.7.23).

## What this is

Phase 2 of the Option C wtf-windows-adoption work. Two coordinated commits:

1. **wtf-windows repo**: adopts `dazzlecmd-lib` as its engine. Deletes 392 LOC of duplicated loader code. Rewrites `cli.py` to use `AggregatorEngine` + `MetaCommandRegistry`. Preserves wtf's domain-specific behaviors (diagnostic badges, mode system, new/add).

2. **dazzlecmd repo (v0.7.24)**: library fixes that surfaced during wtf adoption, plus Phase 3 coordination:
   - `AggregatorEngine(project_root=...)` kwarg so installed aggregators can specify their repo root (library default walks up from the library's own `__file__` which doesn't work for packages installed in site-packages).
   - Python runner package-mode dispatch fix: when a tool directory contains `__init__.py`, the runner now puts the PARENT of the tool dir on sys.path and imports as `<package_name>.<script_stem>` so relative imports in the tool's script resolve correctly.
   - `_cmd_kit_status` uses `_kit_name` over `name` (closes #45).
   - Phase 3: remove `_override_tools_dir` / `_override_manifest` from `kits/wtf.kit.json` (wtf-windows' own kit manifest now declares them via its adopted library).
   - Bump `projects/wtf` submodule pointer.

## High-Value Verification (~5 minutes)

### HV.1 -- wtf standalone CLI works end-to-end

Validates: wtf-windows no longer has its own `loader.py`; library's `AggregatorEngine` powers its CLI; domain-specific rendering preserved.

- [ ] **Steps**:
  ```

  wtf --version
  wtf --help
  wtf list
  wtf info restarted
  wtf kit list
  wtf kit status

  ```

- [ ] **Expected**:
  - `wtf --version` prints "wtf-windows PREALPHA 0.1.3-alpha (...)"
  - `wtf --help` shows the categorized help (diagnostic tools, management, development, examples sections)
  - `wtf list` shows `locked` and `restarted` with `[history, admin]` and `[AI, history]` badges
  - `wtf info restarted` shows runtime fields + Event logs, Privileges, AI support, History
  - `wtf kit list` and `wtf kit status` show wtf's core kit
  - `wtf version` and `--version` print the same string

- [ ] **Why this matters**: wtf's entire CLI was rewritten. If any of these fails, wtf users see a regression.

### HV.2 -- wtf tool dispatch with relative imports

Validates: python runner package-mode fix. Without the fix, tools with `__init__.py` + relative imports fail with "attempted relative import with no known parent package".

- [ ] **Steps**:
  ```
  cd C:/code/wtf-windows
  wtf restarted --help
  wtf restarted history
  ```

- [ ] **Expected**:
  - `wtf restarted --help` prints wtf-restarted's own argparse help (dispatched to the tool's `main()`)
  - `wtf restarted history` runs the history command and prints a table
  - NO "attempted relative import with no known parent package" error
  - NO Python traceback

- [ ] **Why this matters**: the primary bug this commit fixes. Any tool using relative imports (locked, restarted, anything with `__init__.py`) would break without this.

### HV.3 -- 3-tier nesting via dazzlecmd embedded submodule

Validates: `dz wtf:core:locked` dispatches through dazzlecmd's embedded `projects/wtf/` submodule to wtf's core/locked tool. Same tool, different entry path.

- [ ] **Steps**:
  ```
  cd C:/code/dazzlecmd/github
  dz list
  dz info wtf:core:locked
  dz wtf:core:restarted history
  dz tree
  ```

- [ ] **Expected**:
  - `dz list` includes `locked` / `restarted` with Kit="wtf"
  - `dz info wtf:core:locked` resolves correctly (FQCN `wtf:core:locked`, Kit `wtf`, Namespace `core`)
  - `dz wtf:core:restarted history` dispatches through dazzlecmd's engine into wtf's embedded submodule
  - `dz tree` shows `\\-- wtf [aggregator]` with nested tools `wtf:core:locked` and `wtf:core:restarted`
  - NO "attempted relative import" errors (same fix validated through embedded path)

- [ ] **Why this matters**: proves "dazzlecmd is an instance, not the root" — wtf runs standalone AND as an embedded kit. Same library, same codepaths.

## What the automated tests DON'T cover

- **Real subprocess dispatch of wtf tools with relative imports** (mock tests verify the runner constructs the right import; only a real invocation proves the import machinery works)
- **Shell rendering of `wtf` help output in Windows cmd codepage** (ASCII box chars, table alignment)
- **`wtf mode switch` actually flipping submodule state** (destructive; only manual via the `--dry-run` here)
- **3-tier nesting path in a fresh install** (tested on existing checkout with submodule populated)
- **Error messages a user sees when dazzlecmd-lib is missing** (what does `wtf list` say if pip didn't install the dep?)
- **Cross-version compatibility** (wtf pinning `dazzlecmd-lib>=0.1.0` vs the library being at `0.1.0` today; test with older / newer versions)
- **`dazzlecmd-lib` as an installed PyPI package** (currently editable-install; tests from pip-installed package might surface different import paths)
