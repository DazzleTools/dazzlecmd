# Human Test Checklist Addendum: Phase 4b Step 3 — Templates, `dz setup`, Lifecycle

**Target version:** v0.7.13 (Phase 4b Step 3)
**Companion checklist:** `v0.8.0__Phase4b__dazzlecmd-lib-extraction.md` (Steps 1+2)
**Companion automated tests:** 229 existing pytest; no new automated tests for Step 3 (surfaces are thin CLI wrappers + data-file moves)

## What this is

Step 3 of Phase 4b adds three surfaces that the main Phase 4b checklist does not cover:

1. **`dz setup <tool>`** — new meta-command that dispatches a tool's declared `setup.command` (or `setup.platforms.<os>` variant). The engine runs the script; it does not install deps itself.
2. **Templates relocation** — `dazzlecmd.json.tmpl` and `python_tool.py.tmpl` moved from `src/dazzlecmd/templates/` to `packages/dazzlecmd-lib/src/dazzlecmd_lib/templates/`. CLI prefers the library location and falls back to the local `src/dazzlecmd/templates/` if the library is not installed.
3. **`lifecycle` field** — `dz new` scaffolding now writes `{"type": "tool", "status": "active", "created_as": "tool"}` for Phase 5 entity promotion tracking. The library template file mirrors this.

This addendum verifies the three surfaces and the cross-cutting concern that `dz` still works when the library is installed (which it always is post-Step 1+2).

## What mocks systematically miss (Step 3 in particular)

- **Real subprocess dispatch under shell=True**: A.4/A.5 exercise `subprocess.run(shell=True, cwd=tool_dir)` with a real shell. Mocks tend to assert on argv shape and miss shell quoting or cwd confusion.
- **Platform key casing and lookup**: `platform.system().lower()` returns `'windows'`, `'linux'`, `'darwin'` — our code special-cases `darwin` to accept `macos` as an alias. Only a human on each OS catches a regression here.
- **Package data inclusion**: `package_data` in `pyproject.toml` is notoriously easy to get wrong. Pytest tests against the source tree, not the installed wheel. B.1 walks the installed location.
- **Fallback path reachability**: B.3 is a human-eye code read, not an automated test. If we ever delete `src/dazzlecmd/templates/`, this checklist item fails even though tests still pass.
- **Meta-command shadowing**: D.2 is a policy check, not a code check. Adding entries to `meta_commands` is cheap; noticing when one shadows a real tool name is a human concern.
