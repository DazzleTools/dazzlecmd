**Target version:** v0.7.19
**Ships with commit:** TBD (Phase 4c.5)
**Companion automated tests:** 213 new tests across the shared modules and integration (`test_platform_detect.py`, `test_schema_version.py`, `test_paths.py`, `test_conditions.py`, `test_platform_resolve.py`, `test_resolution_trace.py`, `test_user_overrides.py`, `test_resolve_runtime.py`, `test_cli_info.py`). Full suite: 511 passing, 6 platform-skipped.

## What this is

Phase 4c.5 ships conditional dispatch: a single manifest can express different runtime behavior per platform (`runtime.platforms.<os>.<subtype>`) and offer ordered alternatives (`runtime.prefer`). It also lands seven shared library modules (`platform_detect`, `conditions`, `platform_resolve`, `resolution_trace`, `paths`, `schema_version`, `user_overrides`) that the upcoming multi-platform setup feature (issue #40) will reuse.

Automated tests cover the unit-level behavior of each module and the integration resolution pipeline. This checklist covers the human-eyeball surface: actual `dz info` output shape on real tools, error-message readability when nothing matches, and `--platform` preview sanity on hosts you do not own.

## High-Value Verification (~5 minutes)

### HV.1 -- `dz info` default shows resolved runtime for existing tool

Validates: the default info view still works on every existing tool (no regressions to the v0.7.18 baseline).

- [ ] **Steps** (same across all shells):
  ```
  dz info listall
  dz info safedel
  dz info git
  ```

- [ ] **Expected**:
  - Each runs without error, exit code 0.
  - Output shows Name, FQCN, Kit, Runtime, Script/Binary/Interpreter fields appropriate to the tool type.
  - No "resolved for ..." tag appears (none of these tools use conditional dispatch).
  - No traceback, no `NoRuntimeResolutionError`.

- [ ] **Why this matters**: every existing tool's manifest lacks `platforms`/`prefer`, so `resolve_runtime()` should take the fast path and behave identically to v0.7.18. Any regression here breaks every user on upgrade.

### HV.2 -- `--raw` vs default on a conditional-dispatch manifest

Validates: `dz info --raw` prints the declared manifest; default prints the resolved view.

- [ ] **Setup** (manual kit file creation -- `dz kit add` accepts only git URLs, not local paths):

  1. Create a scratch directory structure:

     ```
     test_runs/phase4c-5/
       .kit.json
       preferred-python/
         .dazzlecmd.json
     ```

  2. File `test_runs/phase4c-5/.kit.json`:
     ```json
     {"name": "phase4c-5-scratch", "tools_dir": ".", "manifest": ".dazzlecmd.json"}
     ```

  3. File `test_runs/phase4c-5/preferred-python/.dazzlecmd.json`:
     ```json
     {
         "name": "preferred-python",
         "namespace": "phase4c-5-scratch",
         "runtime": {
             "type": "script",
             "prefer": [
                 {"interpreter": "this-is-not-on-path"},
                 {"interpreter": "python"}
             ]
         }
     }
     ```

     **Note**: `namespace` must match the kit name (`phase4c-5-scratch`),
     not a free-form string. Mismatched namespace makes the tool invisible
     (`_kit_active=False`).

  4. Isolated config via DAZZLECMD_CONFIG:

     **cmd.exe**: `set DAZZLECMD_CONFIG=%TEMP%\phase4c5.json`
     **PowerShell**: `$env:DAZZLECMD_CONFIG = "$env:TEMP\phase4c5.json"`
     **POSIX**: `export DAZZLECMD_CONFIG=/tmp/phase4c5.json`

  5. Activate the kit by editing the config file to include
     `"active_kits": ["phase4c-5-scratch"]`, or register via the library-level
     API. (`dz kit add <path>` doesn't accept local paths; it expects git URLs.
     This is tracked for a future `dz kit add --local` flag.)

- [ ] **Steps** (same across all shells):
  ```
  dz info preferred-python
  dz info preferred-python --raw
  ```

- [ ] **Expected** for default:
  - `Runtime:     script  (resolved for <os>...)` with the current platform tag.
  - `Interpreter: python` (second prefer entry picked, first failed precondition).
  - Note at the bottom: "manifest declares conditional dispatch; use --raw to see the full declaration".

- [ ] **Expected** for `--raw`:
  - `Runtime:     script  (raw, unresolved)`
  - `Prefer:      2 entries (in order)` followed by `[0] interpreter=this-is-not-on-path` and `[1] interpreter=python`.

- [ ] **Why this matters**: the whole point of `--raw` is to let authors see what they wrote vs. what runs. If these views look identical the feature isn't doing its job.

### HV.3 -- `--platform` preview works for a non-current platform

Validates: `dz info --platform <spec>` previews resolution for a platform other than the current host.

- [ ] **Steps** (uses the scratch tool from HV.2):
  ```
  dz info preferred-python --platform linux.debian
  dz info preferred-python --platform windows
  dz info preferred-python --platform macos.macos14
  ```

- [ ] **Expected**:
  - Each run's `Runtime:` line reads `script  (preview for <spec>)` echoing back your platform argument.
  - `Prefer:` section appears with "preconditions not evaluated in preview" and enumerates both entries with their interpreter values.
  - Exit code 0 in each case. No traceback.

- [ ] **Why this matters**: an author on Windows needs to verify their Linux fallback is well-formed without SSH'ing into a Linux box. If `--platform` doesn't reach resolve_platform_block cleanly, this author workflow is dead.

### HV.4 -- Error trace when nothing resolves is readable

Validates: when no `prefer` entry matches on the current host, the error message is actionable.

- [ ] **Setup**: edit the scratch tool's `.dazzlecmd.json` so BOTH entries are guaranteed to fail:
  ```json
  {
      "name": "preferred-python",
      "namespace": "scratch",
      "runtime": {
          "type": "script",
          "prefer": [
              {"interpreter": "not-a-real-cmd-1"},
              {"interpreter": "not-a-real-cmd-2"}
          ]
      }
  }
  ```

- [ ] **Steps** (pick any shell):
  ```
  dz preferred-python
  dz info preferred-python
  ```

- [ ] **Expected**:
  - Running the tool surfaces a `NoRuntimeResolutionError` with a trace that:
    - Lists the current platform (os.subtype, arch).
    - Enumerates each attempted entry with its failure reason (`not on PATH`).
    - Ends with a fix hint ("add a matching `prefer` entry, declare `platforms.<os>.general`, or install the missing interpreter").
  - `dz info` shows `Runtime:     <unresolved for this host>` followed by the same trace content (but without raising -- info never crashes).

- [ ] **Why this matters**: users hitting this error often lack the platform experience to debug silently. The trace should tell them precisely what to do.

### HV.5 -- `--raw` mode on python/script tools shows correct labels

Validates: the runtime-type-aware labels (`Binary:` vs `Script:`, `Shell:`, `NPM script:`, etc.) still render correctly after refactoring into the shared helper.

- [ ] **Steps** (same across all shells, uses existing python tools):
  ```
  dz info restarted --raw
  dz info locked --raw
  dz info git --raw
  ```

- [ ] **Expected**:
  - Each shows `Runtime:     python  (raw, unresolved)` line.
  - Each shows a `Script:` line (not `Binary:`), appropriate for its Python runtime.
  - No `Shell:` label appears (no shell-runtime tool exists in active kits today;
    when one is added, `dz info <shell-tool> --raw` should show the `Shell:` line).
  - If a binary-runtime tool is later added, verify `Binary:` appears instead of `Script:`.

- [ ] **Why this matters**: the `dz info` dispatch-fields helper was extracted into `_print_runtime_dispatch_fields`; a bug in label selection would affect every `dz info` invocation. Eyeballing a few real tools confirms the refactor.

- [ ] **Limitation**: no active kit currently contains a shell-runtime or binary-runtime
  tool, so the `Shell:` and `Binary:` label paths can't be verified with in-repo
  tools. The automated test suite covers these via synthetic manifests in
  `test_cli_info.py::TestPrintRuntimeRaw::test_binary_uses_binary_label`.

## What the automated tests DON'T cover

- **Terminal rendering on Windows codepage**: the `_print_runtime_*` helpers output ASCII characters only, but eyeballing output in cmd.exe / PowerShell catches any codepage issues that mocks don't.
- **Error-message readability**: automated tests assert substrings but humans judge "is this actually helpful?".
- **Performance of the resolution pass**: every `dz <tool>` now goes through `resolve_runtime()`. The fast path (no platforms/prefer) should be essentially free. A slow path here would affect every dispatch; HV.1 implicitly smoke-tests this.
- **`dz info --platform` on exotic specs** (`other`, misspelled subtypes, empty strings): subsequent HV passes can explore these if a regression is reported.
