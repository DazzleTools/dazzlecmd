## What this is

v0.7.22 activates the `user_overrides.py` module that's been groundwork since v0.7.19. Users drop JSON files at `~/.dazzlecmd/overrides/setup/<fqcn>.json` or `~/.dazzlecmd/overrides/runtime/<fqcn>.json` to customize tool behavior without editing the kit's manifest. Deep-merge wins at every scope level; permissive scoping means overrides can add new subtype branches the manifest didn't declare.

## High-Value Verification (~5 minutes)

### HV.1 -- Override a setup command

Validates: user-level override of a setup command without editing the manifest.

**Note**: no currently-active in-repo tool has a `setup.command` declared, so this HV requires EITHER (a) a scratch kit with a setup-declaring tool, OR (b) direct `resolve_setup_block()` Python-API invocation with a synthetic project dict. Option (b) is simpler for a checklist run.

- [ ] **Setup** (option a -- scratch kit): follow v0.7.21 HV.3 procedure to create a scratch kit at `projects/hv22-scratch/alpha-tool/` with a manifest that includes `"setup": {"command": "echo MANIFEST_DEFAULT"}`. Register via `kits/hv22-scratch.kit.json` and active_kits.

- [ ] **Setup** (option b -- Python API, faster):
  ```python
  import os
  os.environ["DAZZLECMD_OVERRIDES_DIR"] = "/tmp/dz-hv1-overrides"
  # Create the override file
  import pathlib, json
  override_dir = pathlib.Path("/tmp/dz-hv1-overrides/setup")
  override_dir.mkdir(parents=True, exist_ok=True)
  (override_dir / "testkit__alpha-tool.json").write_text(
      json.dumps({"_schema_version": "1", "command": "echo HV1_OVERRIDE_FIRED"})
  )
  # Resolve with override
  from dazzlecmd_lib.setup_resolve import resolve_setup_block
  project = {
      "name": "alpha-tool", "_fqcn": "testkit:alpha-tool", "_dir": "/tmp",
      "setup": {"command": "echo MANIFEST_DEFAULT"},
  }
  result = resolve_setup_block(project)
  print(result["command"])  # should print: echo HV1_OVERRIDE_FIRED
  ```

- [ ] **Expected**:
  - With override file: `echo HV1_OVERRIDE_FIRED` (override won)
  - Without override file (delete it): `echo MANIFEST_DEFAULT` (manifest default)

### HV.2 -- Override adds a new subtype branch (permissive scoping)

- [ ] **Setup**: write to `~/.dazzlecmd/overrides/setup/<some-tool-fqcn>.json`:
  ```json
  {
      "platforms": {"linux": {"general": {"command": "echo HV2_FROM_OVERRIDE"}}}
  }
  ```
  (Even if the tool's manifest has no `platforms.linux.general` -- override adds it.)

- [ ] **Steps** (on Linux, or mock the platform):
  ```
  dz setup <some-tool-fqcn>
  ```

- [ ] **Expected**: on a Linux host with a subtype the manifest doesn't cover (e.g. Arch, NixOS), the override's `general` branch fires.

### HV.3 -- Runtime override with `_vars`

- [ ] **Setup**: pick a python-runtime tool; write
  `~/.dazzlecmd/overrides/runtime/<fqcn>.json`:
  ```json
  {
      "_vars": {"python_cmd": "/opt/my-python/bin/python3.12"},
      "interpreter": "{{python_cmd}}"
  }
  ```

- [ ] **Steps**:
  ```
  dz info <fqcn>
  ```

- [ ] **Expected**: resolved Interpreter shows the custom path (template substituted). Note that `dz <tool>` will FAIL because the interpreter path doesn't exist -- that's fine; the point is the override was loaded and `_vars` substituted.

### HV.4 -- Missing override file = no change

Validates: fast path preserved when no override is present.

- [ ] **Steps** (assuming no override file for some tool):
  ```
  dz info <tool-fqcn>
  ```

- [ ] **Expected**: identical output to pre-v0.7.22 behavior. No "override loaded" annotation, no performance change.

### HV.5 -- Malformed override surfaces clean error

- [ ] **Setup**: write garbage to an override file:
  ```
  echo "{not valid json" > ~/.dazzlecmd/overrides/setup/<fqcn>.json
  ```

- [ ] **Steps**:
  ```
  dz setup <fqcn>
  ```

- [ ] **Expected**:
  - stderr shows a JSON-decode error with the path
  - Exit 1, no Python traceback
