## What this is

This document is a **hand-runnable checklist** for verifying that Phase 3 behaves correctly end-to-end from a user's perspective. Automated tests validate the functions and mocked dispatch; this checklist validates the full CLI invocation path, the config file state on disk, shell-level interactions, and the edge cases that are hard to mock.

Every item has:

- **Steps**: what to type
- **Expected**: what you should see
- **Check**: the disk/file state to verify afterward

Work through each section in order — later sections depend on earlier sections having run. After each destructive test, the checklist tells you how to reset state.

## Prerequisites

1. You are on Windows, WSL, or Linux/macOS. Shell-specific commands below are given in THREE forms — **cmd.exe**, **PowerShell**, and **POSIX (bash/zsh)** — pick the matching one for your shell.
2. You have dazzlecmd v0.7.11 installed. Verify: `dz --version` reports `0.7.11`.
3. You have the dazzlecmd source tree checked out somewhere reachable, because some tests use `projects/wtf` as a real nested aggregator.
4. **You MUST run the isolation step below before any other test.** Without it, the checklist commands will write to your REAL `~/.dazzlecmd/config.json` (`%USERPROFILE%\.dazzlecmd\config.json` on Windows).

### Step P.1 — Set the isolation env var (REQUIRED)

This test uses a throwaway config file so your real preferences aren't touched. Pick the form matching your shell:

**cmd.exe (Windows)**:
```cmd
set DAZZLECMD_CONFIG=%TEMP%\dz-phase3-test.json
echo %DAZZLECMD_CONFIG%
```

**PowerShell (Windows)**:
```powershell
$env:DAZZLECMD_CONFIG = "$env:TEMP\dz-phase3-test.json"
Write-Host $env:DAZZLECMD_CONFIG
```

**POSIX (bash/zsh on Linux/macOS/WSL/Git Bash)**:
```bash
export DAZZLECMD_CONFIG=/tmp/dz-phase3-test.json
echo $DAZZLECMD_CONFIG
```

**Verify isolation is active**: the `echo`/`Write-Host` line must print a real path (e.g., `C:\Users\you\AppData\Local\Temp\dz-phase3-test.json`), NOT the literal `%DAZZLECMD_CONFIG%` or `$DAZZLECMD_CONFIG` (which means the var isn't set).

If the variable printed literally, **stop and fix your shell** before continuing. Otherwise every test below will modify your real config.

### Step P.2 — Verify the throwaway config doesn't exist yet

**cmd.exe**:
```cmd
if exist %DAZZLECMD_CONFIG% del %DAZZLECMD_CONFIG%
```

**PowerShell**:
```powershell
if (Test-Path $env:DAZZLECMD_CONFIG) { Remove-Item $env:DAZZLECMD_CONFIG }
```

**POSIX**:
```bash
rm -f "$DAZZLECMD_CONFIG"
```

### Reset between sections

Any of these work from any shell (they all go through `dz`):

```
dz kit reset --yes
```

Or manually delete the file:

| Shell | Command |
|---|---|
| cmd.exe | `del %DAZZLECMD_CONFIG%` |
| PowerShell | `Remove-Item $env:DAZZLECMD_CONFIG` |
| POSIX | `rm -f "$DAZZLECMD_CONFIG"` |

### Viewing the config file

| Shell | Command |
|---|---|
| cmd.exe | `type %DAZZLECMD_CONFIG%` |
| PowerShell | `Get-Content $env:DAZZLECMD_CONFIG` |
| POSIX | `cat "$DAZZLECMD_CONFIG"` |

Or cross-platform via Python (works in all shells if Python is on PATH):

```
python -m json.tool %DAZZLECMD_CONFIG%         # cmd.exe
python -m json.tool $env:DAZZLECMD_CONFIG      # PowerShell
python -m json.tool "$DAZZLECMD_CONFIG"        # POSIX
```

### If you accidentally wrote to your real config

Symptom: you ran a `dz kit` command without setting `DAZZLECMD_CONFIG` first. Your real `~/.dazzlecmd/config.json` (`%USERPROFILE%\.dazzlecmd\config.json` on Windows) now has changes from the checklist.

Recovery options (in order of destructiveness):

1. **Undo the specific change** — e.g., if you ran `dz kit enable wtf`, run `dz kit disable wtf`. The config file is merge-semantic, so targeted undoes work.
2. **Wipe everything** — `dz kit reset --yes`. This deletes the file entirely. Safe if you had no preferences configured before.
3. **If you had important preferences** — sorry, `_write_user_config` doesn't keep backups. You'll need to hand-rebuild the config from memory. This is why Step P.1 is required.

Set the env var now, verify with Step P.1's echo test, and only then proceed to the High-Value Verification section below.

## High-Value Verification (~5 minutes)

**If you only have 5 minutes**, run these six tests. They cover the highest-impact user-visible behavior Phase 3 introduces. If any of them fail, the phase is broken and the detailed sections below probably can't save you.

Each test here is also covered in detail later in this document, but these are the "did the phase actually ship" smoke tests.

**Before running ANY High-Value test**, confirm Step P.1 (isolation env var) has been run. If `echo %DAZZLECMD_CONFIG%` / `Write-Host $env:DAZZLECMD_CONFIG` / `echo $DAZZLECMD_CONFIG` prints a real file path, you're safe. If it prints the literal variable name, STOP and go back to Prerequisites.

### HV.1 — Config file gets written on first command

Validates: `_write_user_config` actually creates the file, atomic write works, schema version injected.

- [ ] **Steps** (pick your shell):

  **cmd.exe**:
  ```cmd
  if exist %DAZZLECMD_CONFIG% del %DAZZLECMD_CONFIG%
  dz kit enable wtf
  type %DAZZLECMD_CONFIG%
  ```

  **PowerShell**:
  ```powershell
  if (Test-Path $env:DAZZLECMD_CONFIG) { Remove-Item $env:DAZZLECMD_CONFIG }
  dz kit enable wtf
  Get-Content $env:DAZZLECMD_CONFIG
  ```

  **POSIX (bash/zsh)**:
  ```bash
  rm -f "$DAZZLECMD_CONFIG"
  dz kit enable wtf
  cat "$DAZZLECMD_CONFIG"
  ```

- [ ] **Expected:** The file exists. Its contents include `"_schema_version": 1` and `"active_kits": ["wtf"]`. Output from `dz kit enable wtf` is `"Enabled kit: wtf"`.
- [ ] **Why this matters:** Every other Phase 3 write command depends on this path. If it's broken, NOTHING in Phase 3 works.

### HV.2 — Kit disable hides tools from `dz list`

Validates: `get_active_kits` honors `disabled_kits`, the discovery pipeline respects config, `dz list` reflects it.

- [ ] **Steps** (same across all shells, uses `dz` commands only):

  ```
  dz kit reset --yes
  dz list
  ```

  Note how many tools are listed (the footer shows "N tool(s) found"). Then:

  ```
  dz kit disable dazzletools
  dz list
  ```

- [ ] **Expected:** The second `dz list` shows fewer tools. No dazzletools tools (dos2unix, split, github, claude-cleanup, etc.) appear.
- [ ] **Why this matters:** This is the central Phase 3 user value — "I can control which kits are in my dz." If this doesn't work, the rest is cosmetic.

### HV.3 — Shadowing removes a tool completely

Validates: `shadowed_tools` filter in `_discover_aggregator`, `dz list` absence, dispatch error, short-name freeing.

- [ ] **Steps:**

  ```
  dz kit reset --yes
  dz list
  ```

  Eyeball the list — note that `safedel` appears. Then:

  ```
  dz kit shadow core:safedel
  dz list
  dz safedel --help
  ```

- [ ] **Expected:**
  - After `dz kit shadow core:safedel`, `safedel` is absent from `dz list`.
  - `dz safedel --help` produces `"invalid choice: 'safedel'"` (argparse error).
- [ ] **Why this matters:** Shadowing is the "primacy principle" in action — the user can tell `dz` to step out of the way when a tool exists elsewhere. Issue #26's headline feature.
- [ ] **Cleanup**: `dz kit unshadow core:safedel` to restore safedel visibility.

### HV.4 — `dz tree` renders correctly (ASCII + JSON)

Validates: Phase 2's FQCN index is consumable as a data structure, output is Windows-codepage-safe.

- [ ] **Steps:**

  ```
  dz kit reset --yes
  dz tree
  ```

  Eyeball the ASCII output. Then:

  **cmd.exe / PowerShell / POSIX** (pipe through Python's json.tool for validation):

  ```
  dz tree --json
  ```

  Optionally capture and validate:

  **cmd.exe**:
  ```cmd
  dz tree --json > %TEMP%\dz-tree.json
  python -m json.tool %TEMP%\dz-tree.json
  ```

  **PowerShell**:
  ```powershell
  dz tree --json | Out-File $env:TEMP\dz-tree.json
  python -m json.tool $env:TEMP\dz-tree.json
  ```

  **POSIX**:
  ```bash
  dz tree --json | python -m json.tool
  ```

- [ ] **Expected:**
  - ASCII output uses `+--`, `|`, `\--` characters. **Check on Windows cmd.exe** that there are no Unicode box-drawing characters (no `├─`, `└─`, `│`). Those would appear as garbled mojibake.
  - JSON output is valid and parseable — `python -m json.tool` exits cleanly with no error.
  - Both show `core`, `dazzletools`, and `wtf` (if wtf submodule is present) with their tools.
- [ ] **Why this matters:** `dz tree` is the first tool that treats the FQCN index as user-visible data. It's the foundation for scripting and tooling integrations. If the output is malformed, every future dz-on-dz tool will be blocked.

### HV.5 — Favorites write and resolve (dry run)

Validates: `dz kit favorite` command writes to the config, reserved-name rejection, stale-target warning.

- [ ] **Steps** (same across all shells):

  ```
  dz kit favorite testname core:fixpath
  dz kit favorite list core:fixpath
  dz kit favorite ghost ghost:nonexistent
  ```

  Then view the config file (see "Viewing the config file" in Prerequisites for your shell).

- [ ] **Expected:**
  - Step 1 (`favorite testname core:fixpath`): stdout `"Favorite set: testname -> core:fixpath"`.
  - Step 2 (`favorite list core:fixpath`): stderr `"Error: 'list' is a reserved command name..."`, non-zero exit.
  - Step 3 (`favorite ghost ghost:nonexistent`): stderr warning about "target FQCN not found", stdout `"Favorite set: ghost -> ghost:nonexistent"`, zero exit.
  - Config file viewing: `favorites` object contains `testname` and `ghost`, but NOT `list`.
- [ ] **Why this matters:** Favorites are the headline fix for issue #9 (collision disambiguation). The reserved-name check prevents users from shooting themselves in the foot with `dz favorite list`.

### HV.6 — Kit focus preserves `always_active`

Validates: `dz kit focus` correctly understands `always_active` semantics, doesn't nuke core.

- [ ] **Steps:**

  ```
  dz kit reset --yes
  dz kit focus wtf
  dz kit list
  dz list
  ```

- [ ] **Expected:**
  - Output from `dz kit focus wtf` mentions preserved kits (core, dazzletools — both `always_active`).
  - `dz kit list` shows core and dazzletools still `[always active]`, wtf `[enabled]`, and any other non-always-active kits as `[disabled]`.
  - `dz list` still shows core tools (fixpath, rn, find, etc.) after the focus — because core is preserved via `always_active`.
- [ ] **Why this matters:** Focus is the "just show me one thing" UX for kit management (issue #18). If it accidentally disabled core, users would lose access to the fundamental dz tools and might not know how to recover.

## Section 1: Config file creation and schema

### 1.1 First-time config creation

- [ ] **Steps:**
  1. Delete the config file if it exists (see "Reset between sections" in Prerequisites for your shell).
  2. Run: `dz kit enable wtf`
- [ ] **Expected:**
  - Output: `Enabled kit: wtf`
  - No errors
- [ ] **Check:**
  - The config file now exists at the path in `DAZZLECMD_CONFIG`
  - Its content has `"_schema_version": 1`
  - Its content has `"active_kits": ["wtf"]`
  - View with `type`/`Get-Content`/`cat` per your shell (see "Viewing the config file" in Prerequisites)

### 1.2 Schema version stays put across writes

- [ ] **Steps:**
  1. Starting from 1.1's state, run: `dz kit disable dazzletools`
  2. View the config file
- [ ] **Expected:**
  - Output: `Disabled kit: dazzletools`
- [ ] **Check:**
  - `_schema_version` is still `1`
  - Both `active_kits: ["wtf"]` and `disabled_kits: ["dazzletools"]` are present

### 1.3 Hand-editable: unknown keys preserved

- [ ] **Steps:**
  1. Reset: `dz kit reset --yes`
  2. Open the config file in your editor and set its content to:
     ```json
     {
       "_schema_version": 1,
       "kit_precedence": ["core"],
       "my_custom_user_key": "do not touch"
     }
     ```
  3. Run: `dz kit enable wtf`
- [ ] **Expected:**
  - Output: `Enabled kit: wtf`
- [ ] **Check:**
  - `my_custom_user_key` is STILL in the file (not removed)
  - `active_kits: ["wtf"]` is added
  - `kit_precedence: ["core"]` is still there

### 1.4 Malformed config is tolerated

- [ ] **Steps:**
  1. Reset: `dz kit reset --yes`
  2. Write garbage to the config file. Pick your shell:

     **cmd.exe**:
     ```cmd
     echo { not json > %DAZZLECMD_CONFIG%
     ```

     **PowerShell**:
     ```powershell
     "{ not json" | Out-File $env:DAZZLECMD_CONFIG
     ```

     **POSIX**:
     ```bash
     echo "{ not json" > "$DAZZLECMD_CONFIG"
     ```

  3. Run: `dz list`
- [ ] **Expected:**
  - A stderr warning like `"Warning: could not read ..."`
  - `dz list` still runs and shows all tools (defaults applied)
- [ ] **Check:**
  - No crash; the list appears

### 1.5 Reset with confirmation

- [ ] **Steps:**
  1. Populate the config: `dz kit enable wtf`
  2. Run: `dz kit reset` (without `--yes`)
  3. At the prompt, type `n` and press Enter
- [ ] **Expected:**
  - Prompt: `"This will delete ... Continue? [y/N]:"`
  - After `n`: `"Aborted."` and non-zero exit
- [ ] **Check:**
  - The config file still exists with `"active_kits": ["wtf"]`

### 1.6 Reset with --yes actually deletes

- [ ] **Steps:**
  1. Populate: `dz kit enable wtf`
  2. Run: `dz kit reset --yes`
- [ ] **Expected:**
  - Output: `"Config cleared."`
- [ ] **Check:**
  - The config file no longer exists

### 6.5 Shadowing frees the short name (manual test)

## What the automated tests DON'T cover (so you DO need to manually)

1. **Windows codepage behavior**: cmd.exe / PowerShell rendering of `dz tree` — must be eyeballed on Windows
2. **Atomic write under real crash**: hard to test; covered by using `os.replace()` semantics
3. **Git submodule integration**: `dz kit add` invokes real git — tested via mocks in automation, but real git behavior only verifiable manually
4. **File locking / concurrent writes**: not currently handled; manual test would try two `dz kit enable` in parallel
5. **Interactive prompts**: `dz kit reset` without `--yes` — automation covers the `--yes` path only
6. **Shell tab completion**: N/A (not implemented yet)
7. **Real user config interactions**: tests use `DAZZLECMD_CONFIG` for isolation, so the real `~/.dazzlecmd/` path is manual-only
8. **Long-running behavior**: repeated read/write cycles, config file bloat over time, schema migration (not implemented yet)
