# Human Test Checklist: Phase 4c.1 — Binary Runner Polish

**Target version:** v0.7.15
**Ships with commit:** TBD (Phase 4c.1)
**Companion automated tests:** 11 tests in `tests/test_registry.py` (`TestBinaryRunnerPrecedence` x7, `TestBinaryRunnerArgPassing` x2, `TestRunnerRegistryBinaryResolution` x2). Full suite: 240 passing.

## What this is

Phase 4c.1 polishes the `binary` runner: documents the dispatch precedence (4-level), adds `DAZZLECMD_FORCE_DEV=1` env var for active-dev workflows, and updates `dz info` to show `Binary:` / `Dev command:` / `Interpreter:` fields where the old output said `Script:` regardless of runtime type. This checklist covers the human-eyeball surface: actual env var behavior end-to-end, output shape on a real binary tool, error message quality.

No binary-type tools ship in the repo today (all current tools are Python), so this checklist has the tester scaffold a temporary `runtime.type: "binary"` manifest to exercise the code paths.

## High-Value Verification (~3 minutes)

### HV.1 — `dz info` labels binary tools correctly

Validates: the `dz info` relabel lands. `Binary:` replaces `Script:` for binary runtime, and `Dev command:` / `Interpreter:` appear when declared.

- [ ] **Setup**: temporarily edit **any existing tool** (e.g., `projects/dazzletools/listall/.dazzlecmd.json`) by adding `"dev_command": "echo would-run-dev"` into its `"runtime"` block. Also change `"type": "python"` to `"type": "binary"` and `"script_path"` to `"listall.py"` for the test.

- [ ] **Steps** (same across all shells):
  ```
  dz info listall
  ```

- [ ] **Expected**:
  - Output line reads `Runtime:     binary` (not `python`)
  - Output line reads `Binary:      listall.py` (the label is `Binary:`, not `Script:`)
  - Output line reads `Dev command:  echo would-run-dev`
  - No crash, no traceback.

- [ ] **Why this matters**: Users inspecting a Rust/Go/C tool need to see at a glance "this is a binary, and here's the dev fallback." If the label still says `Script:`, the relabeling didn't fire.

- [ ] **Cleanup**: `git checkout -- projects/dazzletools/listall/.dazzlecmd.json` to revert the test edit.

### HV.2 — `DAZZLECMD_FORCE_DEV=1` actually forces dev_command

Validates: the env var short-circuits the normal "binary exists → run it" path and takes the dev branch.

- [ ] **Setup**: scaffold a throwaway binary-type tool manifest. Create `test_runs/phase4c-1/fake-tool/` with a `.dazzlecmd.json`:
  ```json
  {
      "name": "fake-tool",
      "namespace": "scratch",
      "runtime": {
          "type": "binary",
          "script_path": "fake-tool",
          "dev_command": "python -c \"print('FORCE_DEV_RAN')\""
      }
  }
  ```
  Also create an empty `fake-tool` (no `.exe`) file so `os.path.isfile` returns True on the binary path.

  **cmd.exe**:
  ```cmd
  type nul > test_runs\phase4c-1\fake-tool\fake-tool
  ```

  **PowerShell**:
  ```powershell
  New-Item -ItemType File -Force -Path test_runs\phase4c-1\fake-tool\fake-tool | Out-Null
  ```

  **POSIX**:
  ```bash
  touch test_runs/phase4c-1/fake-tool/fake-tool
  ```

- [ ] **Steps** (pick your shell):

  **cmd.exe**:
  ```cmd
  set DAZZLECMD_FORCE_DEV=1
  python -c "from dazzlecmd_lib.registry import make_binary_runner; import json; m=json.load(open('test_runs/phase4c-1/fake-tool/.dazzlecmd.json')); m['_dir']='test_runs/phase4c-1/fake-tool'; runner=make_binary_runner(m); exit(runner([]))"
  set DAZZLECMD_FORCE_DEV=
  ```

  **PowerShell**:
  ```powershell
  $env:DAZZLECMD_FORCE_DEV = "1"
  python -c "from dazzlecmd_lib.registry import make_binary_runner; import json; m=json.load(open('test_runs/phase4c-1/fake-tool/.dazzlecmd.json')); m['_dir']='test_runs/phase4c-1/fake-tool'; runner=make_binary_runner(m); exit(runner([]))"
  Remove-Item Env:\DAZZLECMD_FORCE_DEV
  ```

  **POSIX**:
  ```bash
  DAZZLECMD_FORCE_DEV=1 python -c "from dazzlecmd_lib.registry import make_binary_runner; import json; m=json.load(open('test_runs/phase4c-1/fake-tool/.dazzlecmd.json')); m['_dir']='test_runs/phase4c-1/fake-tool'; runner=make_binary_runner(m); exit(runner([]))"
  ```

- [ ] **Expected**:
  - stdout contains `FORCE_DEV_RAN` (proves dev_command fired)
  - exit code 0
  - The empty `fake-tool` file is NOT attempted — no "cannot execute binary file" error, no OS-level error about bad executable format.

- [ ] **Why this matters**: `DAZZLECMD_FORCE_DEV` is the whole point of the change. If this doesn't work, the feature doesn't ship. Mocks verify the argv shape; only running it proves the env var is actually consulted at runtime.

- [ ] **Re-run without the env var** to confirm normal precedence: the output should NOT contain `FORCE_DEV_RAN` (the runner will now try to execute the empty `fake-tool` file, which may error — that's fine; what matters is dev_command was NOT fired).

### HV.3 — Missing binary produces a clean error

Validates: the error message for the "binary missing, no dev_command" case is readable.

- [ ] **Setup**: edit `test_runs/phase4c-1/fake-tool/.dazzlecmd.json` to REMOVE the `dev_command` field, keeping `script_path: "fake-tool"`.

  Then delete the empty `fake-tool` file:

  **cmd.exe**:
  ```cmd
  del test_runs\phase4c-1\fake-tool\fake-tool
  ```

  **PowerShell**:
  ```powershell
  Remove-Item test_runs\phase4c-1\fake-tool\fake-tool
  ```

  **POSIX**:
  ```bash
  rm test_runs/phase4c-1/fake-tool/fake-tool
  ```

- [ ] **Steps** (same across all shells):
  ```
  python -c "from dazzlecmd_lib.registry import make_binary_runner; import json; m=json.load(open('test_runs/phase4c-1/fake-tool/.dazzlecmd.json')); m['_dir']='test_runs/phase4c-1/fake-tool'; exit(make_binary_runner(m)([]))"
  ```

- [ ] **Expected**:
  - stderr reads `Error: Binary not found: <some-path>/fake-tool`
  - exit code 1
  - No traceback, no noise.

- [ ] **Why this matters**: When a user has a Rust/C tool and hasn't compiled yet AND hasn't declared a dev_command, the error should point straight at the problem. Users should be able to read this and immediately know "compile the binary or add a dev_command."

## What the automated tests DO cover

| Area | Test | Verifies |
|---|---|---|
| Precedence level 1 (FORCE_DEV) | `test_force_dev_overrides_existing_binary` | FORCE_DEV + dev_command → dev_command argv |
| Precedence level 1 fallthrough | `test_force_dev_without_dev_command_runs_binary` | FORCE_DEV set but no dev_command → binary still runs |
| Precedence level 2 (binary exists) | `test_binary_exists_runs_binary` | Real binary dispatches |
| Precedence level 3 (dev_command fallback) | `test_binary_missing_dev_command_fallback` | Missing binary + dev_command → cargo-run-style argv |
| Precedence level 4 (no options) | `test_binary_missing_no_dev_command_errors` | Exit 1 on nothing-to-run |
| FORCE_DEV not set | `test_force_dev_env_not_set_runs_binary` | Normal precedence when env var absent |
| Arg forwarding | `test_args_forwarded_to_binary`, `test_args_forwarded_to_dev_command` | argv passes through to subprocess |
| Registry resolution | `TestRunnerRegistryBinaryResolution` x2 | `binary` type registered, `resolve()` returns callable |

## What the automated tests DON'T cover

- **Real env var read at runtime** (HV.2): the tests mock `subprocess.run`; HV.2 actually sets the env var in a real shell and observes the child process's stdout. This is the only end-to-end confirmation that the `os.environ.get` call is wired correctly.
- **`dz info` output shape on a real tool** (HV.1): tests for `_cmd_info` exist but don't specifically assert on the `Binary:` vs `Script:` label swap. A human eyeball on a real tool is faster and more reliable than writing a regex-based output assertion.
- **Error message quality** (HV.3): mocks verify exit code; humans verify the message is actually readable and actionable.
- **Cross-shell env var propagation**: env var handling differs between cmd, PowerShell, and bash. HV.2 exercises one of them, and the tester picking their shell validates that invocation path.
