# Human Test Checklist: Phase 4c.3 — Node Runtime

**Target version:** v0.7.17
**Ships with commit:** TBD (Phase 4c.3)
**Companion automated tests:** 291 passing + 5 skipped. 28 new tests in `tests/test_registry.py` for node runner + script runner `interpreter_args`.

## What this is

Phase 4c.3 adds a dedicated `node` runtime type for the Node.js / npm / TypeScript ecosystem, with three dispatch modes (`script_path`, `npm_script`, `npx`), five supported interpreters (node, tsx, ts-node, bun, deno), and `interpreter_args` placement between interpreter and script. Script runner also gains `interpreter_args` (unblocks JScript `cscript //Nologo //B` and similar). This checklist verifies real subprocess behavior that mocks can't catch.

## High-Value Verification (~6 minutes)

### HV.1 — `.js` file defaults to node

Validates: script-path mode, default interpreter.

- [ ] **Setup**: create `test_runs/phase4c-3/hello-node/hello.js`:
  ```js
  console.log(`HELLO ${process.argv[2] || ''} ${process.argv[3] || ''}`);
  ```
  And `.dazzlecmd.json`:
  ```json
  {
      "name": "hello-node",
      "namespace": "scratch",
      "runtime": {
          "type": "node",
          "script_path": "hello.js"
      }
  }
  ```

- [ ] **Steps** (requires node in PATH):
  ```
  cd test_runs/phase4c-3
  dz hello-node world foo
  ```

- [ ] **Expected**:
  - stdout: `HELLO world foo`
  - exit code 0

- [ ] **Why this matters**: most common case. If this doesn't work, nothing else will.

### HV.2 — `.ts` file without interpreter fails loudly

Validates: TypeScript detection + fail-loud behavior (Q3 decision).

- [ ] **Setup**: rename `hello.js` to `hello.ts` in the manifest (and file); do NOT set `interpreter`.

- [ ] **Steps**:
  ```
  dz hello-node world
  ```

- [ ] **Expected**:
  - stderr: `Error: TypeScript file 'hello.ts' requires an explicit interpreter. Set runtime.interpreter to one of: tsx, ts-node, bun, deno.`
  - Exit code 1, tool does NOT run

- [ ] **Why this matters**: confirms fail-loud. No hidden auto-detection of the user's preferred TS runner.

### HV.3 — Bun or deno inserts `run` subcommand (skip if neither installed)

Validates: per-interpreter profile dispatch.

- [ ] **Setup**: set `interpreter: "bun"` (or `"deno"`) in the manifest; keep `hello.ts` as script_path.

- [ ] **Steps** (bun):
  ```
  dz hello-node alice
  ```

- [ ] **Expected**:
  - `bun run hello.ts alice` is what runs under the hood
  - stdout from the script (or `HELLO alice ` if using bun-compatible TS)
  - Exit code 0

- [ ] **Verify the subcommand**: temporarily run `python -c` to print the resolved argv (see HV.6 for technique). Should show `bun` at index 0, `run` at index 1.

- [ ] **Why this matters**: dispatch parity with how bun users actually invoke these tools. If the subcommand is missing, `bun hello.ts` would try to execute `hello.ts` as a command not a script.

### HV.4 — `interpreter_args` placed between interpreter (and subcommand) and script

Validates: argv construction with deno permissions or node memory flag.

- [ ] **Setup** (deno variant):
  ```json
  "runtime": {
      "type": "node",
      "interpreter": "deno",
      "interpreter_args": ["--allow-read"],
      "script_path": "hello.ts"
  }
  ```

- [ ] **Steps**:
  ```
  dz hello-node alice
  ```

- [ ] **Expected**:
  - Argv at dispatch: `deno run --allow-read hello.ts alice`
  - Exit code 0 (deno honors the permission)

- [ ] **Why this matters**: deno silently REFUSES most file operations without explicit permissions. If `--allow-read` isn't in the right position (between `run` and the script), deno interprets it as a script argument and denies file access.

### HV.5 — `npx` mode downloads on first use (optional, has network side effect)

Validates: npx dispatch + acknowledges the download-on-first-use behavior.

- [ ] **⚠️ Setup**: modify the manifest:
  ```json
  "runtime": {
      "type": "node",
      "npx": "cowsay"
  }
  ```
  (`cowsay` is a small, well-known npm package used as a benign example.)

- [ ] **Steps**:
  ```
  dz hello-node "Phase 4c.3 works!"
  ```

- [ ] **Expected**:
  - First run: npx downloads `cowsay` (network activity visible) — this is **by design and by manifest author's declaration**
  - Outputs an ASCII cow saying "Phase 4c.3 works!"
  - Subsequent runs are faster (cached)
  - Exit code 0

- [ ] **Why this matters**: confirms the Q2 decision — npx is treated like any other dispatch that may touch the network. User's choice to `dz <tool>` is the trust boundary; no special gate required.

- [ ] **⚠️ Cleanup note**: npx caches packages under `~/.npm/_npx/` or similar. `dz safedel` doesn't clean these; run `npx -y clear-npx-cache` or `npm cache clean --force` if you want to clear it.

### HV.6 — Mutual exclusion error (multiple dispatch modes)

Validates: fail-loud on ambiguous manifest.

- [ ] **Setup**: declare BOTH `script_path` and `npm_script`:
  ```json
  "runtime": {
      "type": "node",
      "script_path": "hello.js",
      "npm_script": "hello"
  }
  ```

- [ ] **Steps**:
  ```
  dz hello-node
  ```

- [ ] **Expected**:
  - stderr: `Error: node runtime for hello-node declares multiple dispatch modes (script_path, npm_script). Set exactly one of: script_path, npm_script, npx.`
  - Exit code 1

- [ ] **Why this matters**: ambiguous manifests should fail at dispatch, not silently pick one mode. User gets a clear fix path.

## Section 1: Script runner `interpreter_args` (Q7 fold-in)

### 1.1 — JScript via cscript

- [ ] **Setup** (Windows only): create a tool manifest with:
  ```json
  {
      "name": "jscript-hello",
      "namespace": "scratch",
      "runtime": {
          "type": "script",
          "interpreter": "cscript",
          "interpreter_args": ["//Nologo", "//B"],
          "script_path": "hello.js"
      },
      "platform": "windows"
  }
  ```
  And `hello.js`:
  ```js
  WScript.Echo("HELLO from JScript: " + WScript.Arguments(0));
  ```

- [ ] **Steps** (cmd.exe):
  ```
  dz jscript-hello world
  ```

- [ ] **Expected**:
  - stdout: `HELLO from JScript: world` (no banner since `//Nologo`, runs in batch mode because `//B`)
  - Exit code 0

- [ ] **Why this matters**: confirms `interpreter_args` places the cscript flags correctly. Without them, cscript prints its banner and may block on errors in interactive mode.

### 1.2 — Perl with taint mode (if perl available)

- [ ] **Setup**: manifest with `interpreter: "perl"`, `interpreter_args: ["-w", "-T"]`, script_path pointing at a simple `hello.pl`.
- [ ] **Steps**: `dz <tool>`
- [ ] **Expected**: perl runs with warnings + taint mode active; script dispatches normally.

## What the automated tests DON'T cover

- **Real TypeScript compilation** (HV.3): mocks don't actually invoke tsx/bun/deno on a real `.ts` file. Only a human running HV.3 confirms the subcommand insertion produces valid TS execution.
- **Deno permission enforcement** (HV.4): mocks verify argv shape; only a real deno run proves `--allow-read` is consumed as a deno flag (not forwarded as a script arg).
- **npx network behavior** (HV.5): mocked tests can't download. Only a real `npx cowsay` confirms the download-on-first-use flow works end-to-end.
- **cscript on Windows** (1.1): mocks verify argv; only a real Windows cmd + WSH invocation confirms JScript actually runs with the declared flags.
