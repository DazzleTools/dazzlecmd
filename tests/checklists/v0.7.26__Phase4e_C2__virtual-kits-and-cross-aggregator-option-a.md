# Human Test Checklist: Phase 4e Commit 2 — Virtual Kits + Cross-Aggregator Option A

**Target version:** v0.7.26
**Ships with commit:** `v0.7.26 feat(lib,kits): phase 4e commit 2 -- virtual kits + cross-aggregator Option A`
**Companion automated tests:** 13 tests in `tests/test_virtual_kits.py` + 874 existing regression tests (887 total passing)

## What this is

First working shipment of virtual kits. The four `dazzletools:claude-*` tools become dispatchable as `claude:cleanup`, `claude:session-metadata`, `claude:sesslog-datefix`, and `claude:view` via the new `kits/claude.kit.json` overlay. Canonical FQCNs and short names continue to work unchanged. Aliases do not pollute short-name resolution (rule 7c).

This checklist verifies behavior that mocks cannot: real subprocess dispatch via alias, Windows codepage rendering, shell semantics, error-message clarity, and the cross-aggregator discovery path with a real nested virtual kit.

## High-Value Verification (~5 minutes)

The four tests that prove the feature shipped. If any fails, Commit 2 is broken and the detailed sections below are academic.

### HV.1 — Alias FQCN dispatch

Validates: `claude:cleanup` resolves to the canonical tool and invokes it. This is the headline feature of Commit 2.

- [ ] **Steps**:
  ```
  dz claude:cleanup --help
  ```
- [ ] **Expected**: usage banner starting `usage: claude-cleanup [-h]` with the tool's own arguments. NOT "Tool not found."
- [ ] **Why this matters**: if alias dispatch fails end-to-end, the virtual-kit feature is non-functional.

### HV.2 — Canonical FQCN still works

Validates: no regression on the direct-canonical path. Virtual kits add aliases; they must not break canonical dispatch.

- [ ] **Steps**:
  ```
  dz dazzletools:claude-cleanup --help
  ```
- [ ] **Expected**: identical usage banner to HV.1 (same tool).
- [ ] **Why this matters**: if canonical breaks, we have to ship a patch before anyone can upgrade.

### HV.3 — Short name still works

Validates: `dz claude-cleanup` (the short name of the canonical) continues to resolve. Aliases do NOT change short-name resolution.

- [ ] **Steps**:
  ```
  dz claude-cleanup --help
  ```
- [ ] **Expected**: same usage banner as HV.1 and HV.2.
- [ ] **Why this matters**: if short names stop working, we broke rule 7c.

### HV.4 — Alias short does NOT resolve (rule 7c)

Validates: `dz view` (the alias short of `claude:view`) is NOT a valid command. Aliases provide prettier FQCNs, not new short-name competitors.

- [ ] **Steps**:
  ```
  dz view
  ```
- [ ] **Expected**: argparse error message `invalid choice: 'view'`. The list of choices should NOT include `view`.
- [ ] **Why this matters**: if `view` resolves, we've accidentally polluted short_index with alias shorts.

### HV.5 — Virtual kit visible in `dz kit list`

Validates: the virtual kit is discovered and rendered.

- [ ] **Steps**:
  ```
  dz kit list
  ```
- [ ] **Expected**: output includes a row `claude    4 tool(s)  [always active]` followed by the description starting "Virtual kit: Claude Code helpers..."
- [ ] **Why this matters**: discoverability — users need to see virtual kits are available.

### HV.6 — All four claude aliases dispatch

Validates: every alias declared in `kits/claude.kit.json` routes correctly.

- [ ] **Steps** (run each in order):
  ```
  dz claude:cleanup --help
  dz claude:session-metadata --help
  dz claude:sesslog-datefix --help
  dz claude:view --help
  ```
- [ ] **Expected**: each prints a usage banner for the corresponding canonical tool (`claude-cleanup`, `claude-session-metadata`, `claude-sesslog-datefix`, `claudeview`).
- [ ] **Why this matters**: one alias might work while another silently doesn't; exercise the full `name_rewrite` map.

## Section 3: Cross-aggregator Option A (virtual kits in nested aggregators)

This section verifies the biggest structural change of Commit 2: virtual kits defined inside a nested aggregator (like wtf-windows) are discovered, rewritten with the parent FQCN prefix, and applied at the root.

### 3.1 Create a virtual kit inside the wtf submodule

**NOTE**: We write this into the wtf submodule's working tree. Clean it up at the end of this section so we don't accidentally commit it upstream.

- [ ] **Steps**:

  **cmd.exe / PowerShell / POSIX** — the file contents are identical, the path command differs by shell:

  Write this JSON to `projects/wtf/kits/wtf-virtual-test.kit.json`:

  ```json
  {
      "_schema_version": 1,
      "name": "wtf-virtual-test",
      "virtual": true,
      "always_active": true,
      "description": "Test: virtual kit inside wtf, should be prefixed to wtf:wtf-virtual-test by dazzlecmd root",
      "tools": ["core:locked"],
      "name_rewrite": {"core:locked": "why-locked"}
  }
  ```

- [ ] **Expected**: file created. No immediate dz output needed.

### 3.2 The nested virtual kit appears from root

- [ ] **Steps**: `dz kit list`
- [ ] **Expected**: the output includes `wtf:wtf-virtual-test    1 tool(s)`. The name has been rewritten with the `wtf:` prefix because the kit lives inside the wtf aggregator.

### 3.3 The alias FQCN dispatches to the nested canonical

- [ ] **Steps**:
  ```
  dz wtf:wtf-virtual-test:why-locked --help
  ```
- [ ] **Expected**: the usage banner for the `locked` tool (`usage: locked.py [-h] ...` or similar wtf-specific output). This confirms the virtual kit at `projects/wtf/kits/wtf-virtual-test.kit.json` was collected during recursive discovery, its `name` and `tools` were prefixed with `wtf:`, and the alias resolved end-to-end to the canonical `wtf:core:locked` tool.

### 3.4 Clean up the test kit

**IMPORTANT**: remove the test manifest so it doesn't accidentally get committed upstream to wtf-windows.

- [ ] **Steps** (prefer `dz safedel` for the recoverable-trash safety net):

  ```
  dz safedel --yes projects/wtf/kits/wtf-virtual-test.kit.json
  ```

  Fallback if safedel is unavailable:

  **cmd.exe**:
  ```cmd
  del projects\wtf\kits\wtf-virtual-test.kit.json
  ```

  **PowerShell**:
  ```powershell
  Remove-Item projects\wtf\kits\wtf-virtual-test.kit.json
  ```

  **POSIX**:
  ```bash
  rm projects/wtf/kits/wtf-virtual-test.kit.json
  ```

- [ ] **Expected**: the alias no longer dispatches — `dz wtf:wtf-virtual-test:why-locked --help` now errors with "invalid choice".

## Section 5: Known gaps (Commits 3 and 4)

These sections document behavior that is INTENTIONALLY BROKEN in v0.7.26 and fixed by later commits. Testers should confirm these are still broken (so we don't accidentally "fix" them in a hacky way that regresses Commit 3's cleaner design).

### 5.1 `dz info` is alias-blind (Commit 3 fix)

- [ ] **Steps**:
  ```
  dz info claude:cleanup
  ```
- [ ] **Expected (v0.7.26)**: `Tool 'claude:cleanup' not found. Use 'dz list' to see available tools.`
- [ ] **Expected after Commit 3 (v0.7.27)**: info output for the canonical tool with a "resolved via alias" provenance banner.

### 5.2 `dz tree` does not render virtual kits (Commit 4 fix)

- [ ] **Steps**:
  ```
  dz tree
  ```
- [ ] **Expected (v0.7.26)**: tree shows `core`, `dazzletools`, `wtf` branches. NO separate `claude [virtual]` branch. Tool count footer reads `20 tools across 3 kit(s)` — no aliases mentioned.
- [ ] **Expected after Commit 4 (v0.7.28)**: separate `[virtual]` branch with `->` arrows to canonicals; footer mentions aliases.

### 5.3 DZ_CANONICAL_FQCN / DZ_INVOKED_FQCN not yet injected (Commit 4 fix)

- [ ] **Steps**: run a tool via alias, inspect its own env (any dispatched tool can do this; using `dz claude:cleanup --dry-run` and checking process env).
- [ ] **Expected (v0.7.26)**: neither `DZ_CANONICAL_FQCN` nor `DZ_INVOKED_FQCN` env vars are set when the tool is invoked. This is a known gap.
- [ ] **Expected after Commit 4**: both env vars are set; `DZ_CANONICAL_FQCN=dazzletools:claude-cleanup`, `DZ_INVOKED_FQCN=claude:cleanup`.

## What the automated tests DON'T cover

- **Real subprocess dispatch**: mocks exercise Python-level resolution; this checklist confirms real `subprocess.run` with alias FQCNs works across Windows shells.
- **Windows codepage rendering**: `dz kit list` output with Unicode characters, long virtual-kit descriptions, terminal width wrap.
- **Real wtf-windows submodule**: tests use synthetic aggregator trees; this checklist exercises the real `projects/wtf/` submodule during Section 3.
- **Error message clarity**: whether the 9a and 9b warnings are readable and actionable to a human.
- **Discoverability**: can a new user look at `dz kit list` and understand what a virtual kit is vs a canonical one?
- **Interaction with existing favorites/silenced/shadowed configs**: light exercise via tests but worth eyeballing with a loaded config.
