## What this is

Phase 4c.2 rewrites the shell runner with a per-shell profile table (`SHELL_PROFILES`) supporting 7 shells (cmd, bash, sh, zsh, csh, pwsh, powershell). Scripting-language interpreters (perl, ruby, lua, etc.) deliberately belong in `runtime.type: "script"` with an `interpreter:` field, not here. Adds three manifest fields: `shell_args` (flag replacement), `shell_env` (env-chain), `interactive` (keep-open or exec hand-off). No existing shell-type tools in the repo, so this is net-new capability â€” the checklist verifies behavior against the real `dazzle_env.cmd` pattern and a handful of hello-world fixtures.

## High-Value Verification (~5 minutes)

### HV.1 â€” `dz info` shows all new shell fields

Validates: the `dz info` display enhancement lands for shell-type tools.

- [ ] **Setup**: create `test_runs/phase4c-2/hello-shell/.dazzlecmd.json` with:
  ```json
  {
      "name": "hello-shell",
      "namespace": "scratch",
      "runtime": {
          "type": "shell",
          "shell": "cmd",
          "shell_args": ["/E:ON", "/V:ON", "/c"],
          "shell_env": {
              "script": "C:\\dazzle_env.cmd",
              "args": []
          },
          "interactive": false,
          "script_path": "hello.bat"
      }
  }
  ```
  And `test_runs/phase4c-2/hello-shell/hello.bat` with one line: `@echo HELLO %1`

- [ ] **Steps** (same across all shells):
  ```
  dz info hello-shell
  ```

- [ ] **Expected output lines**:
  - `Runtime:     shell`
  - `Script:      hello.bat` (or `Binary:` if runtime.type were binary â€” should be Script here)
  - `Shell:       cmd`
  - `Shell args:  /E:ON /V:ON /c`
  - `Shell env:   C:\dazzle_env.cmd`
  - `Interactive:` line is absent (since interactive=false; only shown when truthy)

- [ ] **Why this matters**: users inspecting a shell-type tool need to see at a glance what shell + env setup gets invoked. If labels are wrong or missing, tool authors can't debug dispatch.

### HV.2 â€” The dazzle_env.cmd pattern actually works end-to-end

Validates: the real-world chain `cmd /E:ON /V:ON /c "dazzle_env.cmd HOMEBOX PLZWORK && hello.bat"` dispatches correctly.

- [ ] **Setup**: modify the manifest from HV.1 so `shell_env.script` points at an env script you can observe. Create `test_runs/phase4c-2/hello-shell/env_probe.cmd`:
  ```cmd
  @echo off
  set ENV_PROBE_RAN=yes
  echo ENV_PROBE: set marker
  ```
  Update the manifest's `shell_env.script` to `env_probe.cmd` (relative path works if both env and tool are in the same dir).

  Update `hello.bat`:
  ```cmd
  @echo off
  echo HELLO from tool
  echo ENV_PROBE_RAN=%ENV_PROBE_RAN%
  ```

- [ ] **Steps** (pick your shell):

  **cmd.exe**:
  ```cmd
  cd test_runs\phase4c-2
  dz hello-shell
  ```

  **PowerShell**:
  ```powershell
  cd test_runs\phase4c-2
  dz hello-shell
  ```

- [ ] **Expected output**:
  - `ENV_PROBE: set marker`
  - `HELLO from tool`
  - `ENV_PROBE_RAN=yes`

- [ ] **Why this matters**: this is the END-TO-END test of the Phase 4c.2 grammar. If env vars set by the env script don't propagate to the tool script, the chain is broken. Mocked tests verify argv shape; only running it proves the cmd `&&` operator + `/E:ON /V:ON` flags actually carry env state across.

### HV.3 â€” `interactive: true` keeps cmd shell open

Validates: `interactive=true` uses the profile's `interactive_flag` (`/k` for cmd).

- [ ] **Setup**: change the manifest's `interactive` field to `true` (remove or keep `shell_env` as you like â€” both should work).

- [ ] **Steps** (Windows only for cmd):
  ```cmd
  dz hello-shell
  ```

- [ ] **Expected**:
  - Tool output prints normally (`HELLO from tool`, etc.)
  - **After tool completes, the cmd shell stays open** at a `>` prompt
  - Typing `exit` returns control to dz, which then returns to the caller

- [ ] **Why this matters**: the whole point of `interactive=true` is that the shell persists after the tool runs â€” this is the "agent runs a shell in the middle of a task and keeps using it" pattern. If the shell closes immediately, `interactive` isn't doing its job.

- [ ] **Cleanup**: type `exit` at the sub-shell prompt to return.

### HV.4 â€” `interactive: "exec"` hand-off replaces dz process

Validates: `interactive="exec"` uses `os.execvp` to fully hand off.

- [ ] **Setup**: change `interactive` to `"exec"` (string, not boolean).

- [ ] **Steps**:
  ```cmd
  dz hello-shell
  ```

- [ ] **Expected**:
  - Tool output prints normally
  - Shell stays open (same `/k`-like behavior visible to user)
  - **Technical difference from HV.3**: on POSIX, dz's PID has been REPLACED by the shell's PID. On Windows, Python emulates exec by spawning + exiting, so the PID changes. User-visible behavior is similar to `interactive=true` in most cases.
  - Typing `exit` returns to the grandparent shell (not back to dz â€” dz is gone).

- [ ] **Why this matters**: confirms `os.execvp` is wired correctly. This is the foundation for future agentic-task use cases where dz spawns a managed shell and gets out of the way.

- [ ] **Verify**: on a POSIX system (WSL or Linux CI), `echo $$` before `dz hello-shell` and `echo $$` inside the shell should show **different PIDs** (dz's PID is gone; the shell's PID is the replacement).

### HV.5 â€” Unsupported shell_env combinations fail loudly

Validates: the fail-loud-not-silent decision (Q4) works.

- [ ] **Setup**: change manifest's `shell` to `perl` (perl is NOT in the shell profile table â€” it's a scripting-language interpreter, not a shell).

- [ ] **Steps**:
  ```
  dz hello-shell
  ```

- [ ] **Expected**:
  - Error to stderr: `Error: Unknown shell 'perl' for hello-shell. Supported shells: bash, cmd, csh, powershell, pwsh, sh, zsh. For scripting-language interpreters (perl, ruby, lua, etc.), use runtime.type: "script" with interpreter: "perl" instead.`
  - Exit code 1
  - Tool script does NOT run
  - Error message explicitly points the user at `script` runtime type

- [ ] **Also verify**: change `shell` back to something valid (e.g., `cmd`) but set `interactive: true` on `sh`:
  ```json
  "shell": "sh",
  "interactive": true
  ```
  Expected: `Error: interactive mode not supported for shell 'sh'. Supported interactive shells: cmd, bash, zsh, pwsh, powershell`

- [ ] **Why this matters**: users should get immediate, actionable errors for unsupported configurations. Silent ignoring of `shell_env` on perl would mean the tool runs without its env setup, producing confusing downstream failures.

## What the automated tests DON'T cover

- **Real dazzle_env.cmd** on this machine (HV.2): mocks verify argv shape but not the actual cmd.exe `&&` operator + `/E:ON /V:ON` behavior under real Windows cmd. Only a human running HV.2 confirms the chain propagates env state to the tool.
- **`interactive=true` keep-open behavior** (HV.3): subprocess.run with `/k` blocks until user `exit`s. Automated tests can't wait for user input; a human typing `exit` is the only end-to-end validation.
- **`os.execvp` process replacement on POSIX** (HV.4): Python tests can mock execvp but can't verify actual PID replacement. Requires a POSIX terminal.
- **Platform-specific shell availability**: conftest auto-skip handles absent shells, but which shells are present on which CI runner is only visible in actual CI runs.
