# Human Test Checklist: v0.7.28 — Sectioned `dz list` (Option O) + Qualified-Alias Dispatch

**Target version:** v0.7.28
**Ships with commit:** `v0.7.28 feat(cli): phase 4e v0.7.28 -- sectioned dz list (Option O) + qualified-alias dispatch`
**Companion automated tests:** 934 total (+13 new sectioned-render tests, +14 path-form convergence tests)

## What this is

Display refinement on top of v0.7.27. The flat-table layout is replaced with **sectioned rendering** (Option O): tools group under section headers per kit; virtual kits get an annotation `(virtual kit '<name>')` and render adjacent to their canonical parent kit. The flat table is preserved as a fallback when only one section would render (`-k <kit>` filter, single-kit aggregator, etc.).

Also adds **qualified-alias dispatch** so that the form shown in section headers (`dazzletools:claude:cleanup`) is invocable, not just decorative. All five user-typed forms of a tool now converge on the same canonical project.

## High-Value Verification (~5 minutes)

### HV.1 — Default `dz list` is sectioned

- [ ] **Steps**: `dz list`
- [ ] **Expected**: output organized into sections with headers `core:`, `dazzletools:`, `dazzletools:claude:  (virtual kit 'claude')`, `wtf:core:`. Each section has its tools indented two spaces under the header. One blank line between sections.
- [ ] **Why this matters**: the Option O headline feature.

### HV.2 — Virtual-kit section adjacent to canonical parent

- [ ] **Steps**: confirm the order of section headers in `dz list` is:
  1. `core:`
  2. `dazzletools:`
  3. `dazzletools:claude:  (virtual kit 'claude')`
  4. `wtf:core:`
- [ ] **Expected**: `dazzletools:claude:` sits IMMEDIATELY after `dazzletools:`, NOT alphabetically at the bottom.
- [ ] **Why this matters**: the visual extension/parent relationship the user requested.

### HV.3 — Qualified-alias dispatch works

- [ ] **Steps**:
  ```
  dz dazzletools:claude:cleanup --help
  ```
- [ ] **Expected**: usage banner for `claude-cleanup` (the canonical tool). Same as `dz claude:cleanup --help`.
- [ ] **Why this matters**: the form the section header advertises (`dazzletools:claude:`) is now invocable.

### HV.4 — All five forms converge

- [ ] **Steps**: each command should produce identical usage output:
  ```
  dz dazzletools:claude-cleanup --help    # canonical FQCN
  dz dazzletools:claude:cleanup --help    # qualified alias (NEW)
  dz claude:cleanup --help                # short alias FQCN
  dz claude-cleanup --help                # canonical short
  dz cleanup --help                       # alias short (rule 7c)
  ```
- [ ] **Expected**: all five print the same `usage: claude-cleanup [-h] ...` banner.
- [ ] **Why this matters**: the convergence invariant — every visible rendering of the tool resolves to the same canonical.

### HV.5 — `dz info` provenance banner for qualified alias

- [ ] **Steps**: `dz info dazzletools:claude:cleanup`
- [ ] **Expected**: first line of output:
  ```
  (qualified alias 'dazzletools:claude:cleanup' = 'claude:cleanup' -> canonical 'dazzletools:claude-cleanup')
  ```
- [ ] **Why this matters**: distinguishes qualified-alias resolution from short-alias resolution; user sees the path traversed.

### HV.6 — `--show canonical` legacy view still works

- [ ] **Steps**: `dz list --show canonical`
- [ ] **Expected**: sectioned output but WITHOUT virtual-kit sections. All canonical claude-* tools (`claude-cleanup`, `claude-session-metadata`, etc.) visible under `dazzletools:`.
- [ ] **Why this matters**: scripts pinned to `--show canonical` continue to work.

### HV.7 — Single-section fallback to flat layout

- [ ] **Steps**: `dz list -k claude`
- [ ] **Expected**: flat table (Name + Kit + Description columns visible, NO `claude:` section header). Four alias rows.
- [ ] **Why this matters**: when there's only one section to render, the v0.7.27-style flat layout is cleaner.

## Section 3: Qualified-alias dispatch

### 3.1 Qualified form resolves correctly

- [ ] **Steps**:
  ```
  dz dazzletools:claude:cleanup --help
  dz dazzletools:claude:session-metadata --help
  dz dazzletools:claude:sesslog-datefix --help
  ```

- [ ] **Expected**: each prints the corresponding canonical tool's help banner.

- [ ] **Steps**: `dz core:claude:cleanup`
- [ ] **Expected**: argparse "invalid choice" error. The prefix `core` doesn't match the canonical's actual kit `dazzletools`, so qualified-alias resolution fails (correctly).

### 3.3 `dz info` distinguishes qualified vs short alias

- [ ] **Steps**:
  ```
  dz info dazzletools:claude:cleanup
  dz info claude:cleanup
  ```
- [ ] **Expected**: First command's banner reads `(qualified alias 'dazzletools:claude:cleanup' = 'claude:cleanup' -> canonical 'dazzletools:claude-cleanup')`. Second reads `(resolved via virtual-kit alias 'claude:cleanup' -> 'dazzletools:claude-cleanup')`.
