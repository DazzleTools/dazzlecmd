## What this is

First commit where virtual kits are fully first-class citizens: they contribute to short-name resolution, display as distinct branches, surface by default in `dz list`, and inject canonical identity into tool env vars. Closes the discoverability gap raised in `notes/cli/2026-04-20__04-39-32__both_virtual-kit-alias-discoverability-gap.md`.

This checklist verifies the user-facing surfaces (display, dispatch, env vars) end-to-end in real terminals — especially Windows codepage rendering and interactive prompts that mocks can't exercise.

## High-Value Verification (~5 minutes)

### HV.1 — Rule 7c relaxation: `dz cleanup` resolves to the canonical

- [ ] **Steps**: `dz cleanup --help`
- [ ] **Expected**: usage banner for `claude-cleanup` (the canonical tool), NOT "invalid choice". The alias short `cleanup` populates `short_index` now.
- [ ] **Why this matters**: the headline benefit of virtual kits — short names become invokable without needing favorites.

### HV.2 — Alias-blindness audit: `dz info claude:cleanup` works

- [ ] **Steps**: `dz info claude:cleanup`
- [ ] **Expected**: tool metadata with the provenance banner `(resolved via virtual-kit alias 'claude:cleanup' -> 'dazzletools:claude-cleanup')` as the first line, then `Name: claude-cleanup ...`
- [ ] **Why this matters**: v0.7.26 reported "Tool not found" for this input. Commit 3 (fixed in this release) routes info through `engine.find_project` which is alias-aware.

### HV.3 — `dz list` default shows aliases instead of their canonicals

- [ ] **Steps**: `dz list`
- [ ] **Expected**: output includes 4 alias rows `claude:cleanup`, `claude:session-metadata`, `claude:sesslog-datefix`, `claude:view` with Kit column `claude:dazzletools` (Option I format — virtual first, canonical second). The CANONICAL `claude-cleanup` etc. should NOT appear in this default view (they're replaced by aliases). Other canonicals (like `find`, `rn`) still appear.
- [ ] **Why this matters**: virtual kits are invisible in `dz list` on v0.7.26. This is the discoverability fix.

### HV.4 — `dz list --show canonical` preserves legacy view (script stability)

- [ ] **Steps**: `dz list --show canonical`
- [ ] **Expected**: output includes `claude-cleanup`, `claude-session-metadata`, `claude-sesslog-datefix`, `claudeview` rows with canonical short names — no aliases shown. Matches pre-v0.7.27 `dz list` output.
- [ ] **Why this matters**: scripts parsing `dz list` pin to `--canonical` for stable behavior.

### HV.5 — `dz list -k claude` shows the virtual kit's aliases

- [ ] **Steps**: `dz list -k claude`
- [ ] **Expected**: 4 rows (`cleanup`, `session-metadata`, `sesslog-datefix`, `view` in the Name column; `claude:dazzletools` in the Kit column). Footer indicates "4 tool(s) (0 canonical + 4 virtual-kit alias(es))".
- [ ] **Why this matters**: on v0.7.26, `dz list -k claude` returned "No tools found". This is the filter-side alias-blindness fix.

### HV.6 — `dz tree` shows the `claude [virtual]` branch

- [ ] **Steps**: `dz tree`
- [ ] **Expected**: last branch is `\-- claude [virtual, always_active]` followed by four lines:
  ```
      +-- claude:cleanup -> dazzletools:claude-cleanup
      +-- claude:session-metadata -> dazzletools:claude-session-metadata
      +-- claude:sesslog-datefix -> dazzletools:claude-sesslog-datefix
      \-- claude:view -> dazzletools:claudeview
  ```
  Footer: `20 tools across 3 kit(s), 4 alias(es) in 1 virtual kit(s)`
- [ ] **Why this matters**: the primary visual discovery surface finally reflects virtual kits as first-class organizational units.

### HV.7 — `dz kit list claude` drill-in shows alias FQCNs + canonical targets

- [ ] **Steps**: `dz kit list claude`
- [ ] **Expected**: header `Kit: claude [virtual, always active]`, description line, then a table with columns `Alias FQCN | -> Canonical | Description`. Four rows.
- [ ] **Why this matters**: on v0.7.26 this command showed canonical short names and hid the alias namespace. Now drill-in reveals what the virtual kit actually does.

## Section 4: Env var injection (DZ_CANONICAL_FQCN / DZ_INVOKED_FQCN)

### 4.1 Verify env vars are set during dispatch

**Steps** — create a quick env-dumper tool and invoke it both ways. For a fast check without scaffolding a new tool, inspect the library source:

- [ ] `type packages\dazzlecmd-lib\src\dazzlecmd_lib\engine.py | findstr DZ_CANONICAL_FQCN` (cmd)
- [ ] `grep -n DZ_CANONICAL_FQCN packages/dazzlecmd-lib/src/dazzlecmd_lib/engine.py` (POSIX)
- [ ] **Expected**: at least 2 sets of the env var (the registry path and the escape-hatch path).

### 4.2 Env vars restored after dispatch

- [ ] **Expected**: after any `dz ...` invocation completes, the parent shell's `DZ_CANONICAL_FQCN` and `DZ_INVOKED_FQCN` are NOT set (unless you explicitly set them before invocation).
- [ ] **cmd.exe**:
  ```cmd
  set | findstr DZ_CANONICAL
  dz cleanup --help >nul 2>&1
  set | findstr DZ_CANONICAL
  ```
- [ ] **PowerShell**: `Get-ChildItem env:DZ_CANONICAL* -ErrorAction SilentlyContinue`
- [ ] **POSIX**: `env | grep DZ_CANONICAL`
- [ ] **Expected**: no output either before or after the `dz` invocation. Env vars leak only INTO the tool's subprocess, not back to parent.

## Section 5: Stale favorite warning

### 5.1 Create a stale favorite

- [ ] **Steps**: edit your isolated config to include a stale favorite:

  **cmd.exe**:
  ```cmd
  echo {"_schema_version": 1, "favorites": {"ghost": "gone:forever"}} > "%DAZZLECMD_CONFIG%"
  ```

  **PowerShell**:
  ```powershell
  '{"_schema_version": 1, "favorites": {"ghost": "gone:forever"}}' | Out-File -Encoding ASCII $env:DAZZLECMD_CONFIG
  ```

  **POSIX**:
  ```bash
  echo '{"_schema_version": 1, "favorites": {"ghost": "gone:forever"}}' > "$DAZZLECMD_CONFIG"
  ```

### 5.2 Warning emits on any `dz` invocation

- [ ] **Steps**: `dz list 2>&1 | head -2` (POSIX) or `dz list 2>&1 | more` (cmd — press space)
- [ ] **Expected**: first line of output includes `dz: warning: 1 stale favorite(s) detected: 'ghost' -> 'gone:forever'. ...`

### 5.3 `DZ_QUIET=1` suppresses

- [ ] **cmd.exe**: `set DZ_QUIET=1 && dz list 2>&1 | findstr "stale"`
- [ ] **PowerShell**: `$env:DZ_QUIET = "1"; dz list 2>&1 | Select-String "stale"`
- [ ] **POSIX**: `DZ_QUIET=1 dz list 2>&1 | grep stale`
- [ ] **Expected**: no output — warning suppressed.

### 5.4 Cleanup

- [ ] **Steps**: reset config:
  ```
  echo {} > "%DAZZLECMD_CONFIG%"        (cmd)
  echo "{}" > "$DAZZLECMD_CONFIG"       (POSIX)
  ```
