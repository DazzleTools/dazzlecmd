# Human Test Checklist: Phase 4b — dazzlecmd-lib Extraction

**Target version:** v0.8.0
**Ships with commit:** TBD (Phase 4b)
**Companion automated tests:** 201 existing + TBD new tests for ConfigManager, RunnerRegistry, direct library imports

## What this is

Phase 4b extracts the dazzlecmd engine into a separate `dazzlecmd-lib` library package at `packages/dazzlecmd-lib/`. This is a major internal refactor — the engine, loader, config, and runner factories all move to a new package. The dazzlecmd package becomes a thin CLI layer with backwards-compat shims. This checklist verifies that the extraction didn't break anything and that the new library surfaces work correctly.

## Prerequisites

1. Both packages installed in editable mode:

   ```
   pip install -e packages/dazzlecmd-lib/
   pip install -e .
   ```

2. Config isolation for any write tests:

   **cmd.exe**: `set DAZZLECMD_CONFIG=%TEMP%\dz-4b-test.json`
   **PowerShell**: `$env:DAZZLECMD_CONFIG = "$env:TEMP\dz-4b-test.json"`
   **POSIX**: `export DAZZLECMD_CONFIG=/tmp/dz-4b-test.json`

3. Verify isolation is active before write operations.

**Reset**: `dz kit reset --yes` or delete the config file.

## High-Value Verification (~5 minutes)

### HV.1 — Direct library import works

Validates: the extracted library is independently importable.

- [ ] **Steps** (cross-platform Python one-liner):
  ```
  python -c "from dazzlecmd_lib.engine import AggregatorEngine; print('OK')"
  python -c "from dazzlecmd_lib.registry import RunnerRegistry; print(RunnerRegistry.registered_types())"
  python -c "from dazzlecmd_lib.config import ConfigManager; print('OK')"
  python -c "from dazzlecmd_lib.loader import discover_kits; print('OK')"
  ```
- [ ] **Expected:** All four print without ImportError. Registry shows `{'python', 'shell', 'script', 'binary'}`.
- [ ] **Why this matters:** If library imports fail, no third-party aggregator can use the engine. This is the whole point of the extraction.

### HV.2 — Backwards-compat shim works

Validates: existing `from dazzlecmd.engine import ...` paths still work.

- [ ] **Steps:**
  ```
  python -c "from dazzlecmd.engine import AggregatorEngine; print('shim OK')"
  python -c "from dazzlecmd.loader import resolve_entry_point; print('shim OK')"
  python -c "from dazzlecmd.loader import _make_python_runner; print('legacy alias OK')"
  ```
- [ ] **Expected:** All three succeed. No ImportError. The `_make_python_runner` legacy alias maps to the public `make_python_runner` from the registry.
- [ ] **Why this matters:** If shims break, the 201 existing tests fail and all user code that imports from `dazzlecmd.*` breaks.

### HV.3 — Same class identity through both paths

Validates: the shim re-exports the SAME objects, not copies.

- [ ] **Steps:**
  ```
  python -c "from dazzlecmd.engine import AggregatorEngine as A; from dazzlecmd_lib.engine import AggregatorEngine as B; print(f'Same class: {A is B}')"
  ```
- [ ] **Expected:** `Same class: True`
- [ ] **Why this matters:** If they're different classes, isinstance checks fail and objects created via one path don't interoperate with code expecting the other.

### HV.4 — dz CLI end-to-end through extraction layer

Validates: the full CLI works after the engine moved to the library.

- [ ] **Steps** (same across all shells):
  ```
  dz --version
  dz list
  dz tree --depth 1
  dz kit list
  ```
- [ ] **Expected:** All four produce correct output. `dz --version` shows the current version. `dz list` shows 19 tools. `dz tree` shows 3 kits. `dz kit list` shows enabled/disabled status.
- [ ] **Why this matters:** If CLI dispatch is broken by the extraction, users can't use dz at all.

### HV.5 — RunnerRegistry dispatches a real tool

Validates: the registry-based dispatch works end-to-end.

- [ ] **Steps** (needs wtf enabled — use clean config):

  **cmd.exe**: `set DAZZLECMD_CONFIG=nul`
  **PowerShell**: `$env:DAZZLECMD_CONFIG = "NUL"`
  **POSIX**: `export DAZZLECMD_CONFIG=/dev/null`

  Then:
  ```
  dz locked --version
  dz fixpath --help
  ```
- [ ] **Expected:** `dz locked --version` prints wtf-locked version (goes through `make_subprocess_runner` module mode). `dz fixpath --help` prints fixpath help (goes through `make_python_runner` flat mode).
- [ ] **Why this matters:** If the registry doesn't dispatch correctly, tools crash or silently fail.

### HV.6 — ConfigManager works standalone

Validates: the extracted ConfigManager reads and writes independently of the engine.

- [ ] **Steps:**
  ```python
  python -c "
  import os, json
  os.environ['DAZZLECMD_CONFIG'] = os.path.join(os.environ.get('TEMP', '/tmp'), 'cm-test.json')
  from dazzlecmd_lib.config import ConfigManager
  cm = ConfigManager()
  cm.write({'test_key': 'test_value'})
  data = cm.read()
  print(f'schema_version: {data.get(\"_schema_version\")}')
  print(f'test_key: {data.get(\"test_key\")}')
  os.unlink(os.environ['DAZZLECMD_CONFIG'])
  print('Cleanup OK')
  "
  ```
- [ ] **Expected:** `schema_version: 1`, `test_key: test_value`, `Cleanup OK`.
- [ ] **Why this matters:** If ConfigManager doesn't work standalone, no third-party aggregator can use config features.

## What the automated tests DON'T cover

1. **Direct `dazzlecmd_lib.*` imports** — all tests go through `dazzlecmd.*` shims. No test verifies the library is independently importable.
2. **RunnerRegistry standalone** — no test creates a registry, registers a type, and resolves it. All dispatch goes through the full engine pipeline.
3. **ConfigManager standalone** — no test uses `ConfigManager` directly (tests use `engine._get_user_config()` which delegates).
4. **`set_manifest_cache_fn` callback** — no test verifies the hook injection works.
5. **Class identity across paths** — no test checks `AggregatorEngine` from both import paths is the same class.
6. **Library has no CLI imports** — no test verifies the library doesn't accidentally import from `dazzlecmd.*`.
7. **`meta_commands` configurable parameter** — no test passes a custom meta_commands set to the engine.
8. **Third-party aggregator simulation** — no test creates an engine from `dazzlecmd_lib` without dazzlecmd installed.
9. **Performance** — is the shim indirection measurably slower? (Probably not, but not measured.)
